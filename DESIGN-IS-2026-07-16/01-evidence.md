# 阶段 3 真实主应用设计证据

## 运行时证据

- 本地 Docker 前后端 `18080/18090` 可访问，真实登录和权限接口成功。
- 已覆盖驾驶舱、合同、客户发票、客户回款、供应商、供应商付款 6 个桌面页面，以及驾驶舱、合同、回款 3 个移动宽度页面。
- `/web/projects/list` 当前为 404，项目页面源码存在但没有正式动态菜单。
- 客户发票页面显示的金额与驾驶舱口径相差约一万倍；移动回款表隐藏金额、项目和状态。
- 供应商付款为空，但页面未解释数据源、前置关系和导入路径。
- 运行时证据存放于 `.omo/ulw-loop/019f64e6-71eb-75b3-984a-bce75e2ab503/evidence/runtime/runtime-evidence.md`。

## 菜单与路由证据

- 当前可见一级菜单为概览、合同管理、客户财务、供应商管理、供应商财务。
- 项目执行是孤儿域：合同详情会跳转 `ProjectDetail`，但该命名路由未注册。
- `/suppliers/detail` 同时存在动态详情和 Legacy 自重定向，存在冲突。
- 详情页普遍固定返回某个列表，不保留来源、筛选、分页和滚动上下文。
- `ui/` 覆盖层、`ui-vben` 主源码和已构建 `dist` 的路由契约可能漂移。

## 业务语义证据

- 客户合同 45 份、项目 50 个；多数合同缺阶段、交付物或付款计划，项目经理字段大量缺失。
- 供应商实际付款、付款凭证、阶段付款关联均为 0 条。
- 当前 34 条 `inbound` 发票记录实际来自历史客户回款迁移，却被供应商页面解释为供应商发票。
- 财务快照与发票/回款事件表存在金额差异，当前页面没有对账说明。
- 50 个项目风险均为默认低风险，与另一历史状态表的高风险记录矛盾。
- 驾驶舱 `generated_at` 只是接口响应时间，却显示为“数据更新时间”。

## 页面功能证据

- 合同详情把客户回款继续作为发票类型，并链接到不存在的项目路由。
- 客户回款“自动匹配”会直接写库，无预览、确认、置信度和撤销批次；浏览器验收不得触发。
- 供应商列表传 `supplier_id`，当前详情页却把它当统一社会信用代码。
- 供应商付款创建表单不含后端必填的 `supplier_id`，提交必然失败。
- 多个页面存在“开发中”按钮、无动作入口、错误吞掉或错误接口。
- 未知金额、风险、付款状态等多处被强制变成 `0`、优秀或待支付，破坏内容诚实度。

## Vben 与视觉证据

- 真实核心路由页使用 `Page` 的数量为 0。
- 核心页面同时使用 `useVbenVxeGrid`、原生 VxeTable、Ant `a-table` 和 HTML table。
- 32 个核心源文件约 6901 行，包含 86 个 HEX、10 处渐变、37 处显式阴影、12 个原生 `a-modal`，没有 `prefers-reduced-motion`。
- 项目设计规范中的 `--pm-*` Token 没有落地；页面同时存在 Vben Token、`--dash-*`、Tailwind 固定色、Ant 色名和业务 HEX。
- 多页重复 `min-h-screen` 灰色渐变壳，在 Vben 内容区内制造第二套页面背景。
- 驾驶舱先显示等权 KPI 和图表，管理者行动区被放在后面。
- 多个可点击 `div/span`、整行点击和无焦点环交互不满足键盘与语义要求。
- 图表没有文本摘要或可访问数据表；移动表格通过隐藏关键列造成信息损失。

## 多端证据

- 仓库没有 PM Director 小程序、原生 App、PWA、Service Worker、离线草稿、同步队列或 Push 实现。
- 现有浏览器上传可复用文件与数据库关联模式，但没有拍摄批次、GPS、签名、分片、断点续传、幂等和弱网恢复。
- 桌面 Web 应承担复杂核验、全文比对、批量治理和财务对账；未来小程序适合现场采集与轻量确认，管理者 App 适合摘要、提醒与审批。

## 安全边界

审计发现仓库文档中存在疑似真实 OCR 与企查查凭据。证据未复述。该问题需要单独轮换和历史清理，不能在 UI 改造中静默处理。
