# 阶段 3 真实主应用重设计审计范围

## 审计目标

本次审计对象是 PM Director 当前可运行的 Vben 主应用，不是 `?review=phase3` 评审 Demo。目标是判断现有菜单、驾驶舱和全部核心业务页是否能支持真实合同、项目履约、客户应收、供应商协同与数据核验任务，并据此决定保留、重构或替换的范围。

## 主要用户

- 项目总监：识别需要经营干预的合同、项目、回款和供应商事项。
- 项目经理：推进阶段、交付物、验收证据、责任人与风险处置。
- 财务人员：分别处理客户开票/回款与供应商收票/付款，完成对账和核销。
- 合同与资料运营：从 PDF、OCR DOCX、JSON、低质量表格和飞书记录中核验字段、处理冲突并形成系统主数据。
- 未来移动端现场与管理人员：承担资料采集、轻量确认、风险提醒和审批；当前仓库尚无可发布小程序或 App 工程。

## 核心业务约束

1. 客户应收与供应商应付是两条独立资金链，不能混用记录或统计口径。
2. PDF 合同原件是法律事实源；OCR、JSON、Excel 和飞书记录是派生或候选数据。
3. 数据必须区分已核验、待核验、推断、冲突、缺失、历史快照和数据源未建立。
4. 当前项目风险、进度、合同状态包含默认值，不能未经说明作为经营事实。
5. 不虚构审批、RBAC、离线同步、推送、现场采集或供应商付款闭环。

## 技术边界

- Vben Admin 5、Vue 3、Ant Design Vue、VxeTable、ECharts、FastAPI。
- 后端动态菜单是当前运行时菜单真相。
- 真实页面应使用 Vben `Page`、`useVbenVxeGrid`、Vben Form/Modal/Drawer 和现有 `StateBlock`。
- 所有页面默认使用浅色工作区；不以大块暗色、重渐变或装饰性卡片墙制造“科技感”。
- 生产改动必须按测试先行方式实施，并由真实浏览器完成桌面与移动宽度验收。

## 审计范围

- 菜单、动态路由、Legacy 重定向、详情返回契约。
- 综合驾驶舱与 4 个现有视图。
- 合同列表/详情、项目列表/详情。
- 客户发票列表/详情、客户回款列表/详情。
- 供应商列表/详情、供应商发票列表/详情、供应商付款列表/详情。
- Vben 兼容性、视觉系统、响应式、无障碍、动效和内容诚实度。

## 不在本轮内静默执行的事项

- 修改或清洗生产业务数据。
- 自动匹配回款、批量修正金额、删除历史记录。
- 建设真实小程序、App、审批流或 RBAC。
- 提交 Git 或发布生产环境。
