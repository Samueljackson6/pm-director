# 阶段 3 Demo 设计审计范围

## 审计对象

- `prototypes/phase3-ui-demo/`：上一轮阶段 3 静态交互 Demo。
- `docs/design/阶段3-任务细化与Demo评审方案-20260715.md`：上一轮页面和任务方案。
- `docs/UI设计规范.md`：上一轮 UI 规范。
- 当前生产前端 `ui-vben/apps/web-antd/`：用于核对真实框架、组件和路由能力，不把生产代码本身作为本次视觉评分对象。

## 主要用户

项目总监及项目、合同、财务协作人员，当前仍大量依赖国网合同原件、人工整理表格和人工核验数据完成项目履约与收付款管理。

## 主要任务

从合同原件和低质量表格出发，准确确认客户合同、项目执行、开票回款、供应商采购与付款的关系、状态、来源和下一步行动。

## 审计约束

- 技术栈：Vben Admin、Vue 3、Ant Design Vue、后端动态路由、FastAPI。
- 默认视觉：减少暗色大色域，以浅色高可读工作区为主。
- 数据现实：合同原件和表格仍是主要输入，数据库不是所有字段的源头真值。
- 业务现实：重点验证国网合同类型、验收开票回款特征及供应商关联。
- 多端演进：桌面 Web 之外，必须考虑小程序和 App 的移动任务与 API 复用。
- 实施边界：本次先审计与建模，不修改生产前端、后端或业务数据。

## 参考证据

- 项目知识库、合同规则、数据清洗与审计文档。
- 后端数据库结构与脱敏聚合统计。
- Vben 本地官方文档和项目内已验证组件写法。
- 旧 Demo 的 HTML、CSS、JavaScript、截图与浏览器验收记录。

## 预期输出

1. 旧 Demo 的设计证据、10 项评分与明确 verdict。
2. Vben 组件级兼容矩阵。
3. 国网合同、收付款、供应商关系和数据来源模型。
4. Web、小程序、App 的边界与共享领域模型。
5. 业务事实驱动的新信息架构和 Demo 重做方案。
