## 一、核心设计理念——"合同+项目"双核模型

### 1.1 为什么要引入"项目"概念

当前的 `contracts` 表把"合同"和"项目"混在一起了。实际上：

```
合同 ≠ 项目

合同 = 法律文件（甲方/乙方/金额/签订日期/扫描件）
项目 = 执行单元（交付/付款/人员/进度）

一个合同 → 至少产生 1 个项目（多数1:1）
一个项目 → 可能关联 1+ 个合同（框架合同/多期/补充协议）

它们的交集：
  交付阶段 · 交付时间 · 交付内容 · 付款计划
  （合同约定，项目执行）
```

### 1.2 数据主次关系

```
合同（法律层面）          项目（执行层面）
┌─────────────┐          ┌─────────────┐
│ contract_id │←──1:N──→│ project_id   │
│ party_a/b   │          │ manager      │
│ sign_date   │          │ status       │
│ PDF原件     │          │ progress     │
│ OCR文档     │          │ risk_level   │
│ SGSC编号    │          │ 实际开始/完成 │
└──────┬──────┘          └──────┬──────┘
       │                        │
       │    共享的实体            │
       ├─── stages（合同约定阶段，项目执行追踪）
       ├─── deliverables（合同约定交付物，项目实际交付）
       ├─── payments（合同约定付款，项目实际回款）
       └─── timeline（合同约定时间，项目实际节点）
```

### 1.3 参考的成熟实践

- **Oracle Primavera / 微软 Project**：WBS工作分解、阶段-里程碑-任务三层
- **SAP PS/Microsoft Dynamics**：合同→项目→采购订单的三角关系
- **Procore / 广联达**：工程类项目的文档-阶段-财务联动
- **禅道 / JIRA**：项目和迭代/版本管理

```

│      │                      │                               │
```

### 2.1 合同表 (contracts) — 从现状扩展

原有字段保留，新增：

| 新增字段 | 类型 | 说明 |
|---------|------|------|
| `official_name` | TEXT | 合同正式名称（可能不同于项目名） |
| `sgsc_id` | TEXT | 国网统一编号（从mapping表合并过来） |
| `ocr_doc_path` | TEXT | OCR后的.docx文件路径（飞书云盘或本地） |
| `contract_status` | TEXT | 合同状态：草稿/已签订/执行中/已变更/已终止/已完成 |
| `estimated_amount` | REAL | 预估金额（税率变更前） |
| `tax_inclusive` | BOOLEAN | 是否含税 |
| `change_history` | TEXT | 变更历史（JSON） |

### 2.2 项目表 (projects) — 新建核心表

```sql
CREATE TABLE projects (
    project_id      TEXT PRIMARY KEY,         -- PJ-YYYYMM-NNN
    project_name    TEXT NOT NULL,            -- 项目名称
    customer_name   TEXT,                     -- 客户单位（甲方）
    project_type    TEXT,                     -- 科研类/服务类/物资类/施工类
    project_status  TEXT DEFAULT 'active',    -- active/ paused/ completed/ cancelled
    
    -- 项目金额（汇总关联合同）
    total_contract_amount REAL DEFAULT 0,
    
    -- 项目时间
    planned_start   TEXT,                     -- 计划开始
    planned_end     TEXT,                     -- 计划结束（合同约定交付时间）
    actual_start    TEXT,                     -- 实际开始
    actual_end      TEXT,                     -- 实际结束
    
    -- 人员
    project_manager TEXT,                     -- 项目经理
    tech_lead       TEXT,                     -- 技术负责人
    sales_lead      TEXT,                     -- 销售/商务负责人
    
    -- 进度
    overall_progress REAL DEFAULT 0,          -- 综合进度 0-100
    risk_level      TEXT DEFAULT 'low',       -- low/ medium/ high/ critical
    
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.3 合同↔项目关联表 (contract_project_link)

```sql
CREATE TABLE contract_project_link (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    contract_id     TEXT NOT NULL,
    project_id      TEXT NOT NULL,
    link_type       TEXT DEFAULT 'primary',   -- primary/ supplementary/ framework/ amendment
    link_note       TEXT,
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (contract_id) REFERENCES contracts(contract_id),
    FOREIGN KEY (project_id) REFERENCES projects(project_id)
);
```

### 2.4 合同文件表 (contract_files) — 新建

```sql
CREATE TABLE contract_files (
    file_id         TEXT PRIMARY KEY,         -- FILE-YYYYMMDD-NNN
    contract_id     TEXT NOT NULL,
    file_name       TEXT NOT NULL,            -- 原始文件名
    file_type       TEXT NOT NULL,            -- scan_pdf/ ocr_docx/ amendment/ attachment
    file_path       TEXT,                     -- 存储路径（本地/飞书云盘）
    file_size       INTEGER,                  -- 字节
    upload_time     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    upload_by       TEXT,                     -- 上传人
    is_latest       BOOLEAN DEFAULT 1,        -- 是否最新版本
    version         TEXT DEFAULT 'v1',
    notes           TEXT,
    FOREIGN KEY (contract_id) REFERENCES contracts(contract_id)
);
```

### 2.5 人员表 (personnel) — 新建

```sql
CREATE TABLE personnel (
    person_id       TEXT PRIMARY KEY,         -- PER-YYYYMM-NNN
    person_name     TEXT NOT NULL,
    phone           TEXT,
    email           TEXT,
    department      TEXT,                     -- 销售部/技术部/管理层
    role_tags       TEXT,                     -- 销售/技术/管理（逗号分隔）
    status          TEXT DEFAULT 'active',
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.6 项目人员关联表 (project_personnel)

```sql
CREATE TABLE project_personnel (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id      TEXT NOT NULL,
    person_id       TEXT NOT NULL,
    role_type       TEXT NOT NULL,            -- 项目经理/技术负责人/销售负责人/签订人/财务
    assigned_date   TEXT,
    is_active       BOOLEAN DEFAULT 1,
    FOREIGN KEY (project_id) REFERENCES projects(project_id),
    FOREIGN KEY (person_id) REFERENCES personnel(person_id)
);
```

### 2.7 付款凭证表 (payment_vouchers) — 新建

（对供应商的付款凭证，区别于发票）

```sql
CREATE TABLE payment_vouchers (
    voucher_id      TEXT PRIMARY KEY,         -- VOU-YYYYMMDD-NNN
    supplier_contract_id INTEGER NOT NULL,
    supplier_id     TEXT NOT NULL,
    project_id      TEXT,
    voucher_date    TEXT,
    amount          REAL NOT NULL,
    voucher_no      TEXT,                     -- 凭证编号
    payment_method  TEXT,                     -- 银行转账/现金/承兑
    attachment_path TEXT,                     -- 凭证扫描件路径
    notes           TEXT,
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by      TEXT,
    FOREIGN KEY (supplier_id) REFERENCES suppliers(supplier_id)
);
```

## 三、功能模块设计

### 3.1 合同管理模块

| 功能 | 说明 | 优先级 |
|------|------|:----:|
| **合同台账** | 列表+搜索+筛选（类型/状态/时间/金额）+排序 | P0 |
| **合同详情** | 基本信息+关联项目+阶段+付款+交付物+文件 | P0 |
| **合同新增** | 表单录入（手动）+ 信息校验 | P0 |

| **合同模板** | 科研类/服务类模板预填 | P2 |

### 3.2 项目管理模块

| 功能 | 说明 | 优先级 |
|------|------|:----:|
| **项目台账** | 列表+搜索+状态筛选 | P0 |
| **项目详情** | 基本信息+关联合同+阶段进度+付款进度+交付进度 | P0 |

### 3.3 财务模块

| 功能 | 说明 | 优先级 |

## 八、人员与 RuoYi 融合考虑

### 8.1 两条路径

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| A: 独立人员管理 | pm-director 自建 personnel 表 | 灵活快速，但和RuoYi账号不打通 |
| B: 对接RuoYi用户 | 用RuoYi的 `sys_user` 表作为人员源 | 统一账号，但依赖RuoYi API |

### 8.2 推荐：B为主，A辅助

- **人员基础信息**从RuoYi `sys_user` 读取（登录账号、部门、角色）
- **项目角色**（项目经理/技术负责人/销售负责人）在 pm-director 的 `project_personnel` 表维护
- **当前过渡期**：手动维护 personnel 表，后期对接 RuoYi

## 十、衡量成熟度的标准

一个真正成熟的合同管理系统，不仅要"能看"，还要"能管"、"能防"、"能追"：

| 维度 | 初级 | 成熟 |
|------|:--:|:--:|
| **合同** | 列表+详情 | 生命周期管理（草稿→签订→变更→终止），版本追溯 |
| **项目** | 手动维护进度 | 阶段进度自动推导，延期自动预警 |
| **财务** | 记录开票回款 | 应收应付自动计算，账龄分析，回款预测 |
| **人员** | 记录姓名 | 角色分配+工作量统计，离职移交提醒 |
| **文档** | 存储路径 | 版本管理，OCR关联，全文检索 |
| **关联** | 孤立实体 | 合同↔项目↔阶段↔付款↔发票↔人员全链路可追溯 |
