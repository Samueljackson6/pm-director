## 二、数据库 Schema 成熟度评估

### 2.1 现有表结构评价

| 表名 | 行数 | 成熟度 | 评价 |
|-----|------|-------|------|
| `contracts` | 42 | ⚠️ **膨胀** | 48列，字段混杂（商务+技术+财务+验收+分包混合在一张表） |
| `stages` | 47 | ✅ 良好 | 覆盖科研类阶段所需 |
| `payments` | 21 | ⚠️ 偏低 | 21条记录，覆盖率仅50%合同 |
| `deliverables` | 126 | ✅ 良好 | 数量充足 |
| `invoices` | 87 | ✅ 良好 | 覆盖较好 |
| `finance_records` | 44 | ⚠️ 冗余 | 与 invoices 数据重叠，应统一管理 |
| `contract_clauses` | 229 | ✅ 良好 | 新表，但需从OCR填充 |
| `personnel` | 0 | ❌ 空表 | 无人员数据 |
| `project_personnel` | 0 | ❌ 空表 | 无项目团队关联 |
| `payment_vouchers` | 0 | ❌ 空表 | 无凭证数据 |
| `projects` | 50 | ✅ 有数据 | 但与 contracts 重复度高 |

### 2.2 ⭐ 关键缺失：项目经费预算表

**科研类合同的独特需求** — 每个科研合同都有详细的经费预算表：

```
科目名称      预算金额  甲方拨款  乙方自筹
人工费        83.81     83.81     0
设备使用费     0         0         0
业务费        18.49     18.49     0
材料费         9.29      9.29     0
...
```

**建议新建表** `contract_budgets`：

| 列名 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自动递增 |
| contract_id | TEXT | FK → contracts |
| category | TEXT | 科目（人工费/设备费/业务费等） |
| total_budget | REAL | 预算总额 |
| party_a_funding | REAL | 甲方拨款 |
| party_b_funding | REAL | 乙方自筹 |
| notes | TEXT | 备注 |
| sort_order | INTEGER | 排序 |

### 2.3 建议优化项

| 问题 | 建议 |
|------|------|
| `contracts` 表 48 列过于臃肿 | 拆分为：`contracts`(核心信息) + `contract_services`(服务详情) + `contract_acceptance`(验收) + `contract_bidding`(中标) + `contract_subcontract`(分包) |
| `finance_records` vs `invoices` 数据重叠 | 以 `invoices` 为主表，`finance_records` 改为视图 |
| `personnel`/`project_personnel` 空表 | OCR 中有联系人信息，可批量导入 |
| `payment_vouchers` 空表 | 后期功能扩展时启用 |
| `projects` 与 `contracts` 1:1 关系 | 合并或清晰区分 project 与 contract 的边界 |
