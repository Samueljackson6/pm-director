## 二、处理流程（当前）

### 2.1 合同入库流程

```
步骤1：人工打开 OCR 后 .docx 文件
步骤2：人工提取关键字段（项目名称、甲方乙方、金额、付款计划等）
步骤3：编写标准数据 JSON → 存入 cache/contracts/
步骤4：导入 JSON 到 SQLite 数据库（scripts/import_to_db.py）
步骤5：验证数据完整性和准确性
```

### 2.2 PDF 扫描件的定位

| 用途 | 说明 |
|------|------|
| **原始存档** | 飞书云盘保留完整PDF，作为法律效力原件 |
| **留档备查** | 236服务器本地保存PDF副本，方便对照查阅 |
| **审计使用** | 需要核验原始合同内容时，查看PDF原件 |
| **系统展示** | 合同管理系统中展示PDF原件（需在236搭建文件服务） |

### 2.3 PDF 副本存放方案

PDF 扫描件存放于 236 服务器路径 `/home/samuel/OCdoc/合同PDF原件/`，合同管理系统通过静态文件服务或单独的文件服务接口展示。

## 三、与旧流程的差异

| 维度 | 旧流程（已废弃） | 当前流程 |
|------|----------------|----------|
| **数据源** | PDF 扫描件 | OCR 后 .docx |
| **提取方式** | PP-StructureV3 / 百度OCR 自动识别 | 人工阅读 + AI 辅助提取 |
| **准确率** | 83.3%（自动识别） | 接近100%（人工确认） |
| **处理时间** | ~67秒/合同（自动） | ~5分钟/合同（人工+AI） |
| **PDF 角色** | 识别对象 | 存档备查 |
| **工具需求** | PP-StructureV3、百度OCR | 文本编辑器 + 提取脚本 |

### 3.1 OCR 工具保留说明

PP-StructureV3 等 OCR 工具**保留但不作为日常使用**，仅在以下场景启用：
- 新增合同且尚未手动 OCR
- 需要对 PDF 局部内容进行快速查阅
- 人工提取的数据需要交叉验证
