## 三、付款计划 vs 阶段详情 — 设计决策

### 现状分析
- `payments` 表：21条记录，部分 "payment_stage" 与 `stages.stage_name` 有对应关系
- `stage_payment_link` 表：38条记录，但仅 SGSC 合同有数据
- 前端已有付款进度卡片（阶段列表+金额），但数据不全

### 设计：融合展示

**阶段详情表（已有）+ 对应付款信息列**

每阶段行右侧增加：
```
┌─────────────────────────────────────────────────────────┐
│ 阶段      │ 阶段内容 │ 时间 │ 状态 │ 对应付款             │
│──────────│─────────│──────│──────│─────────────────────│
│ 研究5.1  │ 调研收资 │ 6月  │ 已完成 │ ¥30.86万 ✅ 已付   │
│ 研究5.2  │ 算法设计 │ 7-9月│ 进行中 │ ¥46.24万 ⏳ 未付   │
│ 研究5.3  │ 模型优化 │ 10月 │ 待开始 │ —                    │
└─────────────────────────────────────────────────────────┘
```

**付款进度卡片（保留，作为独立视图）**
- 用于展示完整付款计划：阶段/条件/金额/状态
- 同时补充质保金、验收后付款等独立付款项

> 不强制要求所有阶段都有对应付款。科研类合同往往阶段与付款多对多（多个阶段完成后付一笔款），此时 stage_payment_link 作关联查询。

## 六、执行方案

### 第1步：新建 contract_budgets 表 + 自动提取

```sql
CREATE TABLE contract_budgets (
  id INTEGER PRIMARY KEY,
  contract_id TEXT NOT NULL,
  category TEXT NOT NULL,      -- 科目：人工费/设备费/业务费等
  total_budget REAL,           -- 预算总额
  party_a_funding REAL,        -- 甲方拨款
  party_b_funding REAL,        -- 乙方自筹
  notes TEXT,
  sort_order INTEGER
);
```

从 OCR 提取 9 份科研合同的预算表 + 项目简表。

### 第2步：全量 OCR→DB 更新脚本

编写 `ocr_extract_to_db.py`，对 32 份 OCR 逐一：
1. 解析进度计划 → **新建/更新 stages**（12+ 合同）
2. 解析支付计划 → **新建/更新 payments**
3. 解析服务内容/方式/地点/验收标准 → **更新 contracts**
4. 解析交付物时间 → **更新 deliverables.planned_date**
5. 解析条款原文 → **更新 contract_clauses**
6. 解析预算表 → **新建 contract_budgets**
7. 解析项目简表 → **新建 personnel + 更新 project_leader**

## 七、⚠️ 风险与注意事项

1. **OCR 错字已在 v2 清洗脚本中处理**，但仍有少量残余，提取时需用正则精确匹配
2. **阶段↔付款关系不一定是 1:1**，科研类多为多阶段→一笔付款，`stage_payment_link` 作柔性关联
3. **部分合同 OCR 质量低**（扫描件模糊），提取结果需人工核验
4. **contact 字段过短问题** — 部分联系人只有单字（"张庸"、"张华"），可能原始数据如此，非 OCR 问题
