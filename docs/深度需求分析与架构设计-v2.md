## Part A: 国网合同体系全貌

### A.1 合同类型矩阵（当前+未来）

```
                    ┌─────────────────────────────────────────────────┐
                    │            国网合同类型体系                       │
                    ├───────────┬─────────┬─────────┬─────────────────┤
                    │ 科研类    │ 服务类   │ 物资类   │ 施工类          │
                    │ (13,¥944万)│(27,¥1,441万)│(0)     │(0)             │
                    ├───────────┴─────────┴─────────┴─────────────────┤
                    │ 框架合同 │ 协议库存 │ 其他特殊类型               │
                    │ (0)      │ (0)      │ (0)                        │
                    └─────────────────────────────────────────────────┘
```

### A.2 各类型合同的核心差异

| 维度 | 科研类 | 服务类 | 物资类 | 施工类 | 框架/协议库存 |
|------|--------|--------|--------|--------|---------------|
| **核心产出** | 知识产权（专利/论文/软著） | 服务成果（报告/数据） | 实体物资 | 工程实体 | 多次采购/多项目 |
| **合同周期** | 1-3年，阶段明确 | 月~年，服务期限 | 交货周期 | 施工周期+质保期 | 框架期（1-3年） |
| **金额结构** | 固定总价，阶段付款 | 固定总价/人天 | 单价×数量 | 工程量清单 | 单价协议/预估总量 |
| **进度度量** | 研究阶段完成度 | 服务完成比例 | 交货批次进度 | 工程进度百分比 | 子订单完成度 |
| **付款节奏** | 里程碑付款 | 竣工+质保/分期 | 预付款+到货款+质保金 | 进度款+结算+质保 | 子订单独立结算 |
| **关键风险** | 技术失败风险 | 交付质量风险 | 供货延期风险 | 安全事故风险 | 数量预估偏差 |
| **验收方式** | 专家评审 | 成果验收 | 数量+质量检验 | 竣工验收 | 分批验收 |

### A.3 国网合同编号体系

当前数据中观察到的三种编号并行：
- **ZH02-YYYYMMNNN**：内部项目管理编号
- **SGSC[A-Z0-9]{14}**：国网统一合同编号
- **财务编号**（如 SGSCLINNAJJS2600028）：财务系统编号

它们在系统中应该建立明确的映射关系，而非各自独立。

### B.2 供应商管理维度

| 维度 | 字段 | 现状 | 缺失 |
|------|------|------|------|
| **基本信息** | subcontractor, subcontractor_contact | 部分有 | 统一社会信用代码、联系人 |
| **合同信息** | subcontract_amount, subcontract_date | 部分有 | 合同扫描件、合同条款 |
| **开票状态** | sub_invoice_total, sub_invoice_unbilled | data exists | 发票号、发票日期、发票类型 |
| **付款状态** | sub_payment_total, sub_payment_unpaid | data exists | 付款日期、付款凭证号 |
| **履约评价** | 无 | 无 | 质量评分、延期次数 |
| **合作历史** | 无 | 无 | 历史合作项目、累计金额 |

### B.3 供应商财务模型

```
供应商合同金额 (subcontract_amount)
        │
        ▼
供应商已开票 (sub_invoice_total) ─── 未开票 (sub_invoice_unbilled)
        │
        ▼
我司已付款 (sub_payment_total) ─── 未付款 (sub_payment_unpaid)
                                        │
                                        ▼
                                   应付账款管理
```

关键业务指标：
- **付款进度** = sub_payment_total / subcontract_amount
- **开票进度** = sub_invoice_total / subcontract_amount
- **应付账款周转** = sub_payment_unpaid 的账龄分析

## Part C: 发票管理

### C.1 当前数据中的发票信息

`invoice_history` 字段存储格式（TEXT字段，非结构化）：
```
2024-09-24：180300元
2024-10-18：180300元
2024-11-15：90200元
```

### C.2 发票管理设计

发票是连接**合同→财务→税务**的关键桥梁。

```
发票管理体系:
  ├── 客户开票（我司→甲方）
  │   ├── 已开票金额 (invoice_total)
  │   ├── 未开票金额 (invoice_unbilled)
  │   ├── 开票明细 (date, amount, invoice_no, tax_type)
  │   └── 开票计划 (预计何时开票)
  │
  ├── 供应商收票（供应商→我司）
  │   ├── 已收票金额 (sub_invoice_total)
  │   ├── 未收票金额 (sub_invoice_unbilled)
  │   ├── 收票明细 (date, amount, invoice_no)
  │   └── 抵扣状态
  │
  └── 发票管理操作
      ├── 开票申请 → 审批 → 开具 → 送达 → 验证
      ├── 收票登记 → 验真 → 抵扣 → 归档
      └── 红冲/作废处理
```

### C.3 发票数据结构建议

当前`invoice_history`是TEXT字段，建议结构化：

```sql
CREATE TABLE invoices (
    invoice_id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    invoice_type TEXT,          -- '客户开票' / '供应商收票'
    invoice_no TEXT,            -- 发票号码
    invoice_date DATE,          -- 开票日期
    amount REAL,                -- 金额
    tax_rate REAL,              -- 税率
    tax_amount REAL,            -- 税额
    status TEXT,                -- '已开/已收/已验真/已抵扣'
    received_date DATE,         -- 收票日期
    payment_status TEXT,        -- '已付款/未付款'
    notes TEXT
);
```

## Part D: 数据模型重构

### D.1 当前数据模型的核心缺陷

| 问题 | 影响 | 严重度 |
|------|------|--------|
| contract ↔ stage 只有6个合同关联 | 94%的合同无阶段跟踪 | 🔴 高 |
| stage ↔ payment 未关联 | 无法核算「完成阶段X→收到付款Y」 | 🔴 高 |
| 发票数据在TEXT中 | 无法查询发票明细、趋势 | 🔴 高 |
| supplier 信息分散 | 供应商画像不完整 | 🟡 中 |
| project_status 表为空 | 进度无法计算 | 🟡 中 |
| 合同类型仅2类 | 无法区分物资/施工/框架 | 🟡 中 |
| 项目编号不统一 | ZH02/SGSC/财务编号三套 | 🟡 中 |

### D.2 重构后的数据模型

```
┌──────────────────────────────────────────────────────────┐
│                    contracts (合同主表)                     │
│  contract_id | 项目编号(SGSC) | 财务编号 | project_type   │
│  ├── 科研类: 有stages+deliverables                       │
│  ├── 服务类: 有service_content+period                    │
│  ├── 物资类: 有delivery_schedule+specs                    │
│  ├── 施工类: 有milestones+safety_req                      │
│  └── 框架类: 有price_schedule+sub_orders                  │

        │

        │     stage_id | contract_id | stage_name | content

        │
        ├── payments (付款计划)
        │     payment_id | contract_id | stage_id(关联!)

        │
        ├── deliverables (交付物)
        │     deliverable_id | contract_id | type | name | qty
        │
        ├── finance_records (财务快照 - 保留现有模式)
        │     record_id | project_id | batch_id
        │     invoice_* | payment_* | sub_* | warranty_*
        │
        ├── invoices (发票明细 - NEW!)
        │     invoice_id | project_id | type | no | date
        │     amount | tax | status
        │
        ├── suppliers (供应商主数据 - NEW!)
        │     supplier_id | name | credit_code | contact
        │     total_contract_amount | evaluation | status
        │
        ├── supplier_contracts (供应商合同 - NEW!)
        │     contract_id | supplier_id | amount | date | type
        │
        └── project_status (项目状态 - 激活)
```

## Part E: 财务同步与RuoYi融合方向

### E.1 财务同步方向确认

**源与目标的关系：**
```
飞书主表 (source of truth for financial data)
    │
    ├── 拉取 (每周) → finance_records (增量追加, batch_id)
    │
    ├── 同步方向: 飞书 → DB (单向)
    │   例外: DB修改过的字段写回飞书时需标记审批
    │
    └── 增量维护铁律:
        ✅ INSERT新记录, 不UPDATE已有
        ✅ 追加历史记录, 不覆盖
        ✅ batch_id可追溯
```

**关键设计决策：**
1. 飞书是财务数据源头 — DB是镜像+增强
2. 每周定时拉取 + 差异比较 + 人工确认
3. 供应商数据从finance_records提取 → 独立suppliers表

### E.2 RuoYi-Vben融合方向

**不推荐全面嵌入RuoYi。** 原因：
1. 当前RuoYi部署的是标准版，合同管理模块需要从零开发Java代码
2. 数据和业务逻辑更适合Python生态（OCR/爬虫/分析）
3. 独立面板迭代速度远快于RuoYi定制

**推荐方向：独立面板为主，RuoYi为辅助。**
- 独立面板(Python+Vue3)：核心业务（合同台账、财务仪表盘、阶段进度）
- RuoYi-Vben：权限管理 + 用户登录（通过OAuth2.0单点登录）
- 数据层：SQLite为源 → 定期增量导出到MySQL供RuoYi报表使用
