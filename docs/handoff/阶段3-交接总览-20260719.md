# 阶段 3 交接总览

## 1. 总体目标

阶段 3 的目标不是给现有后台“换皮”，而是在不更换 Vben 技术栈、不破坏既有业务能力、不写生产数据的前提下，完成以下系统性改造：

1. 以项目总监、项目经理、财务、合同管理员和管理层的真实任务重组菜单与页面关系。
2. 从国网客户合同的合同类型、条款、项目履约、验收、开票、回款和供应商协作关系出发重构驾驶舱。
3. 在保留现有功能和数据逻辑的基础上，重新设计全部核心列表、详情、筛选、状态、空态和移动适配。
4. 建立可解释的数据真值：区分合同原件、OCR/DOCX、结构化抽取、低质量表格、数据库镜像和人工核验结果。
5. 为未来小程序和 App 保留共享领域模型与 API 边界，但不复制桌面后台菜单，也不在本阶段虚构尚未实现的移动能力。

## 3. 已完成的真实生产改造

### 3.1 菜单、路由和返回上下文

- 后端动态菜单收敛为“经营工作台、合同履约、客户应收、供应链”四组。
- 项目台账和项目详情恢复为正式路由，不再错误跳转到驾驶舱或合同详情。
- 供应商详情消除自重定向，供应商收票和付款使用真实页面路径。
- 建立内部来源路由校验和列表筛选、分页、Query 的返回恢复契约。
- 修复 Vben 服务端菜单对绝对子路由的拼接错误。
- 自动化 16 项、生产构建 11/11、真实浏览器 8 条任务流曾在 G002 收口时通过。

### 3.2 详情页拆分与领域真相

- 合同详情拆分为页面编排与独立详情组件，保留请求、文件、履约和财务行为。
- 客户发票详情拆分为概览、关联回款、附件和操作组件。
- 供应商详情拆分为主档概览、工商风险、合作与财务组件。
- 客户发票严格限定为 `outbound + 客户开票`；其他方向或类型显示领域冲突。
- 金额缺失显示“待核验”，已知零值保留为 `0.00`；关联金额使用空值合并，避免把已知 0 覆盖掉。
- 供应商聚合只接受 `供应商开票 + inbound`，不再把历史客户回款误计为供应商收票。
- 供应商工商数据优先读取本地归档，缺档时保留主档，并避免在证据不足时生成健康评级。
- 2026-07-19 聚焦验证：客户发票详情 18 项、供应商详情 13 项通过。
- 2026-07-19 交接回归：后端 9 项、Vben 应用 54 项、菜单工具 2 项，共 65 项通过；直接 Vite 生产构建已生成新产物。
- 应用全量类型检查仍被仓库既有的 511 条错误阻断，主要分布在资产、BPM、WMS 等非阶段 3 模块，并包含驾驶舱测试契约缺口；已列为后续 QA 基线债务。
