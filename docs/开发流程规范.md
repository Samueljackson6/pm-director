# 开发流程规范

> 本文档定义 pm-director 项目的标准开发流程、CI 门禁和部署验收规范。
> 来源：系统全量审计报告 `docs/audits/系统全量审计与改造基线-2026-07-13.md`

---

## 1. 标准开发流程

### 1.1 流程图

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           标准开发流程                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Issue 定义问题                                                           │
│     └─ 明确验收标准                                                          │
│                                                                             │
│  2. 设计/RFC（可选但推荐）                                                    │
│     └─ 确认路由、数据和 UI 边界                                               │
│                                                                             │
│  3. 创建分支                                                                 │
│     └─ 从最新 master 建短分支                                                │
│     └─ 命名规范: feat/xxx, fix/xxx, refactor/xxx                            │
│                                                                             │
│  4. Draft PR                                                                │
│     └─ 先展示方案与风险                                                       │
│     └─ 标注 WIP 或 Ready for review                                         │
│                                                                             │
│  5. 实现                                                                    │
│     └─ 编写代码 + 单元测试 + E2E 测试（如涉及路由）                            │
│                                                                             │
│  6. 代码审查                                                                │
│     └─ 至少 1 人审批                                                         │
│     └─ 解决全部审查对话                                                       │
│                                                                             │
│  7. CI 检查                                                                 │
│     └─ 等待所有检查完成并通过                                                 │
│     └─ 禁止在 CI 完成前合并                                                   │
│                                                                             │
│  8. 合并                                                                    │
│     └─ Squash merge 或 Merge commit                                         │
│                                                                             │
│  9. 部署验收                                                                │
│     └─ 验证构建产物版本                                                       │
│     └─ 功能烟测                                                              │
│                                                                             │
│  10. 关闭 Issue                                                             │
│      └─ 确认验收通过后关闭                                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 禁止流程

| 禁止行为 | 原因 | 正确做法 |
|---------|------|---------|
| 先合并、等 CI | CI 失败后代码已进入 master | 等 CI 完成后再合并 |
| 先关 Issue、再验证部署 | 部署失败时 Issue 已关闭 | 部署验收通过后再关闭 |
| 一个 PR 同时改路由、数据、UI 和基础设施 | 变更范围过大，难以审查 | 拆分为多个独立 PR |
| 跳过 Draft PR 直接提交大型改动 | 审查者无法理解设计意图 | 先提 Draft PR 讨论 |

## 2. GitHub 门禁规范

### 2.1 分支保护规则

master 分支已启用以下保护：

| 规则 | 状态 |
|------|------|
| 禁止直接 push | ✅ 启用 |
| 禁止 force push | ✅ 启用 |
| 禁止删除分支 | ✅ 启用 |
| 必须通过 PR | ✅ 启用 |
| 至少 1 个批准审查 | ✅ 启用 |
| 新提交使旧审批失效 | ✅ 启用 |
| 分支必须为最新 | ✅ 启用 |

### 2.2 必需 CI 检查

| 检查项 | 说明 |
|--------|------|
| Python Lint | 后端代码风格检查 |
| Backend Test (pytest) | 后端单元测试 |
| Frontend Test (vitest) | 前端单元测试 |

### 2.3 建议新增的检查项

| 检查项 | 说明 | 优先级 |
|--------|------|--------|
| Frontend Lint | 前端代码风格检查 | P1 |
| Frontend Typecheck | TypeScript 类型检查 | P1 |
| Frontend Build | 前端构建检查 | P1 |
| Route E2E | 路由矩阵端到端测试 | P2 |
| Data Integrity | SQLite 外键完整性检查 | P2 |

## 3. 部署验收规范

### 3.1 本地 Docker 部署

```bash
# 1. 构建后端镜像
docker compose build backend

# 2. 启动后端
docker compose up -d backend

# 3. 构建前端
cd ui-vben/apps/web-antd && pnpm build

# 4. 重启前端容器
docker restart pm-director-frontend

# 5. 验证部署
curl http://localhost:18080/system/auth/get-permission-info
curl http://localhost:18090/web/
```

### 3.2 部署验收清单

| 检查项 | 验证方法 | 通过标准 |
|--------|---------|---------|
| 后端 API 可访问 | `curl http://localhost:18080/api/dashboard/overview` | 返回有效 JSON |
| 菜单定义正确 | 检查 `/system/auth/get-permission-info` | 路径和组件与预期一致 |
| 前端页面可访问 | 浏览器访问 `http://localhost:18090/web/` | 无 404 错误 |
| 核心流程可用 | Playwright 测试 | 登录 → 导航 → 详情 → 返回 |

### 3.3 版本一致性检查

**问题**：审计发现同一台机器同时存在多套版本：
- 新后端进程 (8800)
- 旧容器后端 (18080)
- 旧前端构建
- 旧磁盘菜单定义

**解决方案**：

1. 每次部署前确认容器状态：
```bash
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.CreatedAt}}"
```

2. 构建后端时确认镜像时间：
```bash
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.CreatedAt}}" | grep backend
```

3. 前端构建后确认产物时间：
```bash
ls -la ui-vben/apps/web-antd/dist/index.html
```

## 4. PR 规范

### 4.1 PR 标题格式

```
<type>: <description>

type 可选值:
- feat: 新功能
- fix: 修复 bug
- refactor: 重构
- docs: 文档更新
- test: 测试相关
- chore: 构建/工具相关
```

### 4.2 PR 描述模板

```markdown
## 背景
<!-- 为什么需要这个 PR？关联的 Issue -->

## 修改内容
<!-- 具体改了什么 -->

## 验证
<!-- 如何验证这些改动？测试截图？ -->

## 风险
<!-- 可能的影响范围 -->

## 检查清单
- [ ] 已添加/更新测试
- [ ] 已更新文档（如适用）
- [ ] CI 检查通过
```

### 4.3 PR 大小限制

| 变更行数 | 建议 |
|---------|------|
| < 100 行 | ✅ 理想大小 |
| 100-300 行 | ⚠️ 考虑拆分 |
| > 300 行 | ❌ 必须拆分 |

## 5. Issue 规范

### 5.1 Issue 类型

| 类型 | 前缀 | 说明 |
|------|------|------|
| Bug | `[Bug]` | 线上问题 |
| Feature | `[Feature]` | 新功能需求 |
| Improvement | `[Improvement]` | 优化改进 |
| Documentation | `[Docs]` | 文档相关 |

### 5.2 Issue 内容要求

```markdown
## 问题描述
<!-- 清晰描述问题或需求 -->

## 复现步骤（Bug 适用）
1. 步骤一
2. 步骤二
3. 预期结果 vs 实际结果

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 相关信息
- 环境：
- 版本：
```

## 6. 发布规范

### 6.1 发布前检查

- [ ] 所有 PR 已合并
- [ ] 所有 CI 检查通过
- [ ] 功能测试通过
- [ ] 回归测试通过
- [ ] 文档已更新

### 6.2 版本号规范

使用语义化版本：`MAJOR.MINOR.PATCH`

- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修复

## 7. 回滚规范

### 7.1 何时需要回滚

- CI 检查失败
- 部署后功能不可用
- 发现严重 bug

### 7.2 回滚步骤

```bash
# 1. 查看最近提交
git log --oneline -5

# 2. 回滚到上一个版本
git revert HEAD

# 3. 推送回滚提交
git push origin master

# 4. 重新构建和部署
docker compose build backend && docker compose up -d backend
cd ui-vben/apps/web-antd && pnpm build
docker restart pm-director-frontend
```

## 8. 附录：历史教训

### 8.1 审计发现的问题

| 问题 | 根因 | 教训 |
|------|------|------|
| PR 在 CI 完成前合并 | 无分支保护 | 必须启用分支保护 |
| 部署持续被跳过 | CI 配置错误 | CI 检查对象必须正确 |
| 供应商菜单 404 | 使用绝对路径作为子路径 | 子路由路径必须是相对路径 |
| 详情返回错误页面 | 没有统一返回契约 | 必须实现来源与返回契约 |

### 8.2 防止复发

1. **每次新增页面**：检查路由是否在 `auth.py` 中定义
2. **每次修改详情页**：确认返回逻辑正确
3. **每次合并 PR**：确认 CI 通过后再合并
4. **每次部署**：验收后再关闭 Issue
