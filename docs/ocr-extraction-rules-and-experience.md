# OCR 提取与数据治理经验总结

> 编写日期：2026-07-12
> 基于全天多轮对话中识别、提取、清洗、修复的实践经验

## 一、合同模板识别

合同 PDF 基于三种标准模板，外加非标准格式：

| 模板编号 | DB类型 | 章节结构 | 提取策略 |
|----------|--------|----------|----------|
| `SGTYHT/24-JS-001` | 科研类 | 1~21 章编号 | `_extract_section()` 按数字章节分割 |
| `SGTYHT/23-JS-004` | 服务类 | 1~14 章编号 | `_extract_numbered_section()` |
| `SGTYHT/25-JS-004` | 服务类(含物资) | 1~14 章编号 | 同23-JS-004，额外提取保修/中标 |
| 非标格式(政府采购/产品购销) | 服务类 | 自定义章节 | 仅提取基本信息 |

**模板检测方法**：文件名前 200 字符搜索 `SGTYHT/24-JS-001` 或 `科学技术项目合同`。

**⚠️ 重要：DB中 project_type 可能不准确**。全天发现 11 个合同 DB 标记为科研类但实际是服务模板。最终修正后：服务类32 : 科研类10。**每份合同应以OCR模板而不是DB标记为准**。

## 二、OCR 文本结构

### 2.1 双重结构问题

所有合同的 OCR 文本都存在 **目录(TOC) + 正文** 双重结构：

```
[第1页] 目录
1.主要内容    2
2.预期目标    3
...
[第3页] 正文
1.主要内容
1.1 主要技术内容
...
```

**陷阱**：提取函数如果用 `re.search()` 取第一个匹配，会匹配到目录中的标题（仅含页码，无正文内容）。

**解决方案**：用 `re.findall()` 取 `[-1]`（最后一个匹配），跳过目录。

### 2.2 章节编号格式

| 格式 | 示例 | 存在于 |
|------|------|--------|
| 带空格的 `7. 知识产权` | `7. 知识产权` | 部分服务合同 |
| 无空格 `7.知识产权` | `7.知识产权` | 大部分服务合同(OCR压缩) |
| 中文编号 `第七条` | `第七条 知识产权` | 科研合同正文 |
| TOC页码 `7.知识产权\n3` | `7.知识产权` + `3`(页码) | 目录区域 |

**陷阱**：`\.\s` 要求编号点后有空白字符，但 OCR 文本中经常是 `7.知识产权`（点号后直接中文）。应使用 `\.(?:\s|[\u4e00-\u9fff])`。

### 2.3 页面标记

OCR 中会插入 `[第N页]`、`SGTYHT/24-JS-001...`、`合同编号：XXX` 等页面标记。提取时应过滤这些噪声。

## 三、分类型提取规则

### 3.1 科研类 (SGTYHT/24-JS-001)

| OCR章节 | 目标 | 提取方法 |
|----------|------|----------|
| 1-3. 主要内容/预期目标/指标 | contracts.service_content | 合并提取 |
| 4. 提供的最终成果 | deliverables | 按编号行提取 |
| **5. 进度计划内容及考核目标** | **stages** | 按阶段编号或日期范围分割 |
| **6. 项目经费** | **contract_budgets** | 从表格提取科目+金额 |
| **7. 项目支付计划** | **payments** | 从表格提取条件+金额 |
| 9/12/14/17 条款 | contract_clauses | 按章节提取 |
| 4.2 指定负责人 | contracts.contacts | 正则提取姓名电话 |

**阶段提取要点**：
- 第一个阶段通常是 `合同签订之日—YYYY年MM月DD日`，应补合约签订日期
- 阶段编号(5.1, 5.2)在日期之后出现，不是之前
- 去重：阶段名本身就是日期范围的条目与"研究阶段X.X"条目重叠

**预算表提取要点**：
- 表格结构：`科目名称 | 预算金额 | 甲方拨款 | 乙方自筹`
- "项目经费"行（合计总预算）不应作为独立科目
- `(—)直接费` 和 `合计` 行应识别为汇总行而非科目

**支付计划提取要点**：
- 支付条件位于表格中，条件行之后紧跟金额行
- 提取时应收集条件行前几行的文字，而非单一行

### 3.2 服务类 (SGTYHT/23-JS-004 / 25-JS-004)

| OCR章节 | 目标 | 提取方法 |
|----------|------|----------|
| 1.2 服务内容 | contracts.service_content | 直接提取 |
| 1.3 服务方式 | contracts.service_method | 提取关键词(咨询/现场/驻场) |
| 2.1 服务地点 | contracts.service_location | 直接提取 |
| 2.2 服务期限 | contracts.service_period | 含起止日期 |
| 4.2 **指定负责人** | **party_a_contact, party_b_contact** | 正则提取姓名和电话 |
| 5. 报酬及支付方式 | payments | 按支付阶段解析 |
| 6. 验收 | acceptance_criteria/method/location | 三字段分别提取 |
| 7-11. 条款 | contract_clauses | 按类别分类 |

**⚠️ 4.2 指定负责人是联系人提取的核心来源**：
```
4.2本合同双方分别指定项目负责人如下：
(1)甲方负责人：XXX，电话：XXX；
(2)乙方负责人：XXX，电话：XXX。
```
该模式存在于所有标准服务合同中。

### 3.3 非标准合同

政府合同(政府采购)和产品购销合同：
- 无标准条款结构，不应提取 clauses
- 只需提取基础信息(甲方/乙方/金额/日期)
- service_content 从正文提取

## 四、数据清洗规则

### 4.1 付款清洗
删除包含以下关键词的付款条目：
- `违约金`、`罚款`、`赔偿`（条款中的违约条款）
- `项目经费`（预算合计的误提）
- `相当于`（违约金描述）
- condition 内容过短(<3有效字)
- 金额远超合同金额5倍

### 4.2 付款条件文本清洗
- 去掉行首 `北京振华永创智能科技有限公` 等OCR碎片
- 去掉 `[第N页]` 页码标记
- 去掉 `SGTYHT/XXXX-XX-XXX` 模板编号
- 去掉行首标点和空格

### 4.3 合同类型清洗
按OCR模板文件编号而非DB标记判断类型。DB标记可能有误。

### 4.4 预算"项目经费"处理
"项目经费"在预算表中实际是**合计行**，不应作为独立科目。
应重命名为"预算合计"或直接删除（用各科目总和代替）。

- 每份合同处理时间：约 2-5 分钟（取决于页数）
- 适合后台运行，需主动轮询进度

### 5.2 DB 操作
- 数据库文件映射：`./database/` ↔ `/app/database/`（容器内）
- 运行提取脚本时，可以在宿主机直接用 Python 3.13+ 连接本地 SQLite
- 如后端使用 Docker，提取后需 `docker compose restart backend`

### 5.3 Python 兼容性
- Docker 容器内 Python 3.11 **不支持** f-string 中的反斜杠表达式：`f'{var["key"]}'` 会报错
- 应使用预赋值：`k = var["key"]; f'{k}'` 或 `%s` 格式化
- 宿主机 Python 3.13.12 无此限制

### 5.4 章节边界检测
- 科研合同用 `_extract_section()`：搜索 `N.关键词` → 找到 `N+1.` 作为结束
- 服务合同用 `_extract_numbered_section()`：搜索 `N.关键词` → 找到 `N+1.` 作为结束
- 都需要用最后一个匹配(跳过TOC)
- 边界正则不能用 `\.\s`（要求点后有空格），要用 `\.(?:\s|[\u4e00-\u9fff])`
- fallback 搜索偏移量必须正确处理
