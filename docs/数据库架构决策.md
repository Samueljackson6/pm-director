## 二、SQLite vs MySQL 对比

| 维度 | SQLite | MySQL |
|------|--------|-------|
| **部署复杂度** | ✅ 零配置，文件存储 | ❌ 需独立服务，权限管理 |
| **运维成本** | ✅ 无需维护 | ⚠️ 需要备份、监控、升级 |
| **并发能力** | ⚠️ 写锁粒度大 | ✅ 行级锁，高并发 |
| **数据规模** | ✅ 支持 TB 级（当前 <2MB） | ✅ 无限制 |
| **查询性能** | ✅ 小数据量优秀 | ✅ 大数据量更优 |
| **备份难度** | ✅ 文件复制即可 | ⚠️ 需要 dump 工具 |
| **灾备能力** | ⚠️ 单文件风险 | ✅ 主从复制 |

---

## 三、决策结论

### ✅ 暂不迁移 MySQL

**理由**：

1. **数据规模小**：当前数据量 <2MB，SQLite 完全够用
2. **用户量有限**：单用户/小团队使用，无高并发需求
3. **运维成本**：迁移 MySQL 会增加部署复杂度和运维负担
4. **备份已完善**：已有手动备份机制，可强化为自动备份
5. **稳定性优先**：当前系统已稳定运行，迁移风险大于收益

## 四、SQLite 强化方案

### 4.1 自动备份策略

```bash
# 每日备份（保留 7 天）
0 2 * * * /home/samuel/.openclaw/workspace/pm-director/scripts/backup-database.sh --daily

# 每周备份（保留 4 周）
0 3 * * 0 /home/samuel/.openclaw/workspace/pm-director/scripts/backup-database.sh --weekly
```

### 4.2 备份验证

- 每次备份计算 SHA256 校验和
- 定期（每月）验证备份可恢复性
- 异地备份：定期同步到云存储（可选）

### 4.3 监控告警

- 数据库文件大小异常（突增/骤减）
- 备份失败告警
- 磁盘空间不足告警

## 五、未来迁移条件

**当出现以下情况时，重新评估迁移 MySQL**：

1. 数据量增长超过 1GB
2. 用户数增长超过 50 人
3. 需要复杂的多表关联查询
4. 需要事务支持和数据一致性保证
5. 需要与其他系统集成
