## 二、开发阶段

### 2.1 目录结构

```
pm-director/
├── backend/           # FastAPI 后端代码
│   ├── main.py        # 主入口
│   └── ...
├── ui/                # Vben 自定义页面代码
│   ├── src/
│   │   ├── api/       # API 接口
│   │   ├── views/     # 页面组件
│   │   └── router/routes/modules/  # 路由定义
│   └── ...
├── scripts/           # 自动化脚本
├── docs/              # 文档
├── systemd/           # 236 systemd 服务文件
└── .github/workflows/ # CI 配置
```

### 2.2 开发前准备

```bash
# 本地：拉取最新代码
git pull origin master

# 236：拉取 + 同步当前状态
ssh 236 "cd /home/samuel/.openclaw/workspace/pm-director && git pull origin master"
```

### 2.3 代码规范

| 项目 | 规范 | 检查方式 |
|------|------|---------|
| Python 后端 | PEP8, ruff 检查 | `ruff check backend/` |
| Vben 页面 | 遵循 `docs/Vben开发规范.md` | tsc --noEmit |
| API 文件 | 使用 `requestClient`，禁用 `baseRequestClient` | 人工审查 |
| 路由 | 子路由用 `path: ''` 而非 `path: 'index'` | 人工审查 |

### 2.4 修改原则

- **只修改必要文件** — 禁止顺手重构无关代码
- **保持项目原有代码风格**
- **不要引入新的依赖** — 除非有充分的理由并记录到 docs
- **敏感信息禁止写死** — 密码、Token、API Key 必须通过环境变量或配置读取

## 三、提交阶段

### 3.1 提交前检查清单

在 `git commit` 之前，必须执行以下检查：

- [ ] 所有改动与当前需求直接相关
- [ ] 没有遗留的调试代码（`console.log`、`print`、`pdb`）
- [ ] 没有硬编码的敏感信息
- [ ] Python 代码语法正确（`python3 -c "import ast; ast.parse(open('X').read())"`）
- [ ] 构建是否能通过（在 236 上运行 `scripts/deploy-236.sh build-only`）

### 3.2 提交信息规范

```
<type>(<scope>): <简短描述>

<详细描述（可选）>

<关联issue（可选）>
```

**类型 (type)**：
| 类型 | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| refactor | 重构 |
| docs | 文档更新 |
| ci | CI/CD 变更 |
| style | 代码风格（不影响功能）|
| test | 测试相关 |
| chore | 杂项 |

**范围 (scope)**：
| 范围 | 含义 |
|------|------|
| vben | Vben 前端页面修改 |
| backend | FastAPI 后端修改 |
| docs | 文档 |
| ci | CI/CD |
| scripts | 自动化脚本 |
| systemd | 服务管理 |
| api | API 接口 |

**示例**：
```
feat(vben): 合同列表页添加按项目名称搜索功能
fix(backend): 修复 stage-payment 关联返回空数据问题
docs(workflow): 更新自动化部署流程文档
```

### 3.3 提交流程

```bash
# Step 1: 检查状态
git status
git diff

# Step 2: 暂存
git add <files>

# Step 3: 提交（注意提交信息规范）
git commit -m "feat(scope): 描述"

# Step 4: 推送到 GitHub
git push origin master
```

## 四、CI 验证（GitHub Actions）

### 4.1 触发条件

| 触发事件 | 操作 | 状态 |
|---------|------|------|
| Push to master | 运行完整 CI/CD（lint + 构建检查 + 自动部署到236）| ✅ 已验证 |
| Push to any branch | 运行基本 lint 检查 | ✅ |
| Pull Request | 运行完整 CI | ✅ |
| 手动触发 (workflow_dispatch) | 运行完整 CI/CD | ✅ |

### 4.2 CI 检查项

```
GitHub Actions CI
├── Python Lint (ruff)
│   ├── backend/main.py
│   └── backend/**/*.py
├── UI 文件完整性检查
│   ├── api/*.ts 存在
│   ├── views/**/*.vue 存在
│   └── router/routes/modules/*.ts 存在
├── 提交信息格式检查
└── 构建验证（模拟 Vben build）
```

### 4.3 当前限制

- 236 服务器无公网 IP，GitHub Actions 无法直接 SSH 部署
- CI 仅做代码验证，不执行实际部署
- 部署通过一键脚本完成（见第五章）

## 五、部署阶段

### 5.1 部署架构

```
开发者
  │
  ├── git push → GitHub (代码托管)
  │                 │
  │                 ├── GitHub Actions CI (自动验证)
  │                 │
  │                 └── 代码同步
  │
  └── scripts/auto-publish.sh (一键发布)
        │
        ├── 1. 本地 lint 检查
        ├── 2. git add + commit + push
        └── 3. SSH 到 236
              │
              ├── 4. git pull
              ├── 5. copy ui/ → ui-vben/
              ├── 6. pnpm build
              ├── 7. systemctl restart (前端 + 后端)
              └── 8. 验证 + 审计
```

### 5.2 一键发布

```bash
# 完整流程（推荐）
bash scripts/auto-publish.sh

# 仅部署（代码已推送时）
bash scripts/auto-publish.sh deploy-only

# 仅验证
bash scripts/auto-publish.sh verify

# 仅审计
bash scripts/auto-publish.sh audit
```

### 5.3 手动部署

如果一键脚本不适用，按以下步骤：

```bash
# Step 1: 本地提交并推送
git add -A
git commit -m "feat(scope): 描述"
git push origin master

# Step 2: SSH 到 236 执行部署
ssh 236 "cd /home/samuel/.openclaw/workspace/pm-director && bash scripts/deploy-236.sh all"

# Step 3: 验证
ssh 236 "bash scripts/deploy-236.sh verify"
```

### 5.4 服务管理（236 systemd --user）

```bash
# 后端服务（无需 sudo）
systemctl --user status pm-backend
systemctl --user restart pm-backend
journalctl --user -u pm-backend -f

# 前端服务
systemctl --user status pm-frontend
systemctl --user restart pm-frontend
journalctl --user -u pm-frontend -f

# 统一管理
bash scripts/start-services.sh status
bash scripts/start-services.sh restart
```

## 六、验证阶段

### 6.1 部署验证清单

每次部署后必须执行以下验证：

- [ ] `curl http://localhost:5777/` → HTTP 200
- [ ] `curl http://localhost:8800/api/stats` → HTTP 200
- [ ] JS bundles 包含 contracts/invoices/suppliers
- [ ] 浏览器访问页面无白屏
- [ ] API 数据正常返回

## 七、审计阶段

### 7.1 审计检查项

每个任务完成后必须执行：

```
检查目标完成度（Ralph 方法）：
1. 任务目标关键词是否出现在输出中？
2. 完成度 < 70% → 继续执行
3. 完成度 ≥ 70% → 标记完成
```

### 7.2 审计流程

```bash
# 方法一：自动审计
bash scripts/auto-publish.sh audit

# 方法二：手动检查
# 1. 检查所有文件是否提交
git status

# 2. 检查构建是否正常
curl http://192.168.0.236:5777/

# 3. 检查后端是否正常
curl http://192.168.0.236:8800/api/stats

# 4. 检查 JS bundles
ls /home/samuel/.../dist/js/contracts-*.js

# 5. 检查目标关键词
# 任务要求的功能是否已实现？
```

### 7.3 审计标准

| 维度 | 标准 | 通过条件 |
|------|------|---------|
| 代码 | 无语法错误 | lint 通过 |
| 构建 | 构建成功 | exit code 0 |
| 部署 | 服务运行 | HTTP 200 |
| 功能 | 任务目标达成 | 关键词匹配 |
| 规范 | 遵循工作流 | 流程完整 |
