## 3. 违约责任章节标准结构（9.x 模板）

所有 35 份合同的违约责任均落在第 9 章，结构高度一致：

```
9. 违约责任
  9.1 总则：乙方不履行/履行不符合约定 → 甲方有权要求 继续履行、赔偿损失 或 支付违约金
  9.1.1 逾期交付：每逾期 1 天，支付相当于技术服务报酬 X% 的违约金；逾期超过 N 日，甲方有权单方解除合同
  9.1.2 经催告仍未纠正 → 甲方有权单方解除合同
  9.1.3 验收不通过：退还甲方已付全部款项 + 支付报酬 X% 的违约金
  9.1.4 违反保密义务：承担法律责任 + 支付报酬 X% 的违约金
  9.1.5 因乙方原因解除：停止支付 + 退还全部款项 + 支付报酬 X% 的违约金
```

**核心规律**：违约金一律以 **「技术服务报酬（合同总额）的 X%」** 为计算基数，而非固定金额。这是结构化抽取的关键锚点。

## 4. 违约金比例矩阵（量化，131 个样本）

按「触发情形」归类后的典型比例（占技术服务报酬 %）：

| 触发情形 | 典型比例 | 备注 |
|----------|----------|------|
| 逾期交付 | **1%/天**（个别 0.5%、0.01%） | 最常见；逾期超 N 日（10/60 不等）触发解除权 |
| 验收不通过 | 1% / 10% / 20% | 一律「退还全部款项」 |
| 保密违约 | **10%**（个别 20%） | |
| 解除合同（乙方原因） | 1% / 5% / 20% | 一律「退还全部款项」 |
| 其他（质量要求等） | 5% | |

**全量比例频次**（131 样本）：
`1%:30, 5%:27, 10%:26, 20%:18, 0.5%:12, 0.01%:3, 0.1%:3, 3%:4, 8%:2, 0.67%:2, 0.05%:1, 30%:3`

**启示**：违约金比例离散但可枚举（集中在 0.5%–20%）。DB 应存「比例值 + 计算基数（技术服务报酬）」而非只存描述文本；同时保留原文片段以备审计。

## 6. 规律总结 → 对 DB 设计的启示

1. **乙方单一、甲方多值**：合同主体应拆 `party_a` / `party_b` 字段；乙方可预设默认值。
2. **违约金基数为「报酬」**：结构化存储 `liquidated_damages_rate` + `ld_base='技术服务报酬'`，可枚举比例。
3. **7 类必含 + 2 类可选**：必含条款做必填结构化字段；可选条款做可空字段。
4. **违约责任结构标准化**：可按 `trigger_type`（逾期/验收/保密/解除/其他）建违约子表，每行存 `rate%`、`refund_full`（是否退还全款）、`threshold_days`（逾期解除阈值）。
5. **OCR 噪声治理**：合同编号小写字母、签订日期 `YYYY.M.D` 需归一化清洗后再入库。
6. **原文留痕**：每条结构化条款保留 `source_clause_text` 原文片段，满足审计与回溯。
