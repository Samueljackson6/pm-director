## 第2轮 · 业务价值层

### 每个角色的核心诉求

**项目经理视角：**
- 一眼看清：**哪些合同在执行、进展如何、钱收回来多少**
- 风险预警：**付款逾期、阶段延期、供应商未结**
- 周报自动化：**不用每周手动汇总**

**财务视角：**
- 开票vs回款对比：**谁欠我们钱、我们欠谁钱**
- 历史追溯：**上个月的开票数据还在吗？**
- 合规校验：**已开票 ≤ 合同金额？已回款 ≤ 已开票？**

**老板/决策者视角：**
- 全局看板：**总合同额、总回款率、TOP客户**
- 趋势分析：**月度开票趋势、回款趋势**
- 异常告警：**哪个项目偏离了计划**

### 价值优先级矩阵

```
                    高频使用          低频使用
                   ┌──────────────────────────┐
    高决策价值     │  全局看板 ★★★★★        │  年度分析报告 ★★★  │
                   │  回款率趋势 ★★★★★       │  供应商绩效 ★★★   │
                   ├──────────────────────────┤
    低决策价值     │  合同台账 ★★★★          │  数据质量报告 ★★   │
                   │  阶段进度更新 ★★★★      │  历史归档 ★        │
                   └──────────────────────────┘
```

## 第3轮 · 数据关系层

### 核心数据模型（当前状态）

```
┌──────────────┐       ┌──────────────────┐
│   contracts   │       │  finance_records  │
│   40条        │◄─────►│  40条 (每周快照)  │
│   ¥2,385万    │       │  开票: ¥1,049万   │
└──────┬───────┘       │  回款: ¥707万     │
       │                └──────────────────┘
       │
       ├─────── stages ──────── 66条 (研究阶段)
       ├─────── payments ────── 57条 (付款计划)
       ├─────── deliverables ── 106条 (交付物)
       └─────── project_status ── 0条 (未激活)
```

### 关键数据关系

| 关系 | 当前状态 | 问题 |
|------|---------|------|
| contract → stage | 已关联 (contract_id) | 只有6个合同有阶段数据 |
| contract → payment | 已关联 | 付款条件不完整 |
| contract → deliverable | 已关联 | 交付物名称不规范 |
| contract → finance | 已关联 | 39个有财务数据 |
| finance → batch | 已关联 | 批量导入可追溯 |
| stage → payment | ❌ 未关联 | 无法知道哪个阶段对应哪笔付款 |
| contract → project_status | ❌ 未激活 | 进度无法自动计算 |

### 数据缺口分析

| 缺口 | 影响 | 修复方案 |
|------|------|----------|
| 34个合同无研究阶段 | 无法进行阶段进度跟踪 | 从合同原文手动/LLM补充 |
| 33个合同无付款计划 | 无法预测现金流 | 同上 |
| 38个合同无交付物 | 交付管理缺失 | 同上 |
| stage↔payment 无关联 | 无法按阶段核算 | 从合同特别约定中推断 |
| 供应商数据不完整 | 供应商财务跟踪不准 | 从飞书补充 |

## 第4轮 · 系统约束层

### 技术约束

| 约束 | 详情 | 应对策略 |
|------|------|----------|
| **数据源在SQLite** | 52个标准数据JSON + 1个DB文件 | SQLite作为源数据，MySQL作为管理镜像 |
| **飞书API可用** | lark-cli可调用 | 飞书是财务数据源 |
| **RuoYi-Vben已部署** | Spring Boot + Vue3 + MySQL | 复用其权限/菜单框架 |
| **Python生态成熟** | FastAPI/Optical/LLM | 数据处理用Python |
| **236服务器** | 单机部署 | 无高可用要求 |
| **Node.js v22** | 前端构建工具 | Vue3 + Vite |

### 数据约束

| 约束 | 详情 |
|------|------|
| 合同原文在.docx中 | 已OCR处理，文本可读但结构不一 |
| 财务数据在飞书表格中 | 需通过API获取 |
| 部分合同金额无法自动提取 | 需要人工核对 |
| 研究阶段/付款计划在合同附件中 | 非标准合同正文内容 |
| 项目编号体系不统一 | ZH02/SGSC/财务编号三套并行 |

### 人员约束

| 约束 | 详情 |
|------|------|
| 系统维护者为非技术人员 | 操作要简单、容错 |
| 数据录入需审批 | 不能随意修改 |
| 每周数据更新需可追溯 | batch_id + 历史保留 |

## 第5轮 · 风险分析层

### 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 合同金额不准 | 中 | 高 | 多源校验 (DB ↔ JSON ↔ 人工) |
| 飞书API变更 | 低 | 中 | 同步脚本加版本检测 |
| 数据丢失 | 低 | 极高 | 每日自动备份 |
| 新合同格式不兼容 | 高 | 中 | 提取器支持手动模板配置 |
| 用户拒绝使用 | 中 | 高 | 渐进式交付，MVP先跑通 |

### 关键依赖

```
飞书API可用 ───┬─── 财务同步可用
               │
OCR准确率 ─────┼─── 合同识别完整度
               │
合同附件可用 ──┴─── 阶段/付款数据完整
```

### 容错设计

| 场景 | 处理方式 |
|------|----------|
| 飞书不可用 | 使用上次快照 + 告警 |
| OCR识别失败 | 跳过，人工补充 |
| 金额校验失败 | 标记异常，不阻塞流程 |
| DB损坏 | 从每日备份恢复 |

## 第6轮 · 设计方案层

### 方案A：纯SQLite + FastAPI + 独立Vue3

```
优点: 快速开发, 本地优先, 零网络依赖
缺点: 无权限管理, 无工作流
适合: 展示面板 (Dashboard)
```

### 方案B：纯RuoYi-Vben 深度定制

```
优点: 统一平台, 权限/工作流/报表开箱即用
缺点: 开发周期长, 需要Java
适合: 后台数据管理
```

### 方案C：混合（推荐）

```
独立面板 (FastAPI+Vue3)       RuoYi-Vben (Spring Boot+Vue3)
  ├── 综合仪表盘                ├── 合同管理 (CRUD)
  ├── 合同可视化                ├── 财务录入
  ├── 财务趋势                  ├── 权限控制
  └── 阶段进度                  └── 工作流审批
       │                               │
       └─────────── 数据同步 ───────────┘
                   cron定时
              SQLite ←→ MySQL
```

### UI设计原则

```
1. 信息密度适中 — 不是堆砌数据，而是有层次的信息呈现
2. 颜色语义化 — 绿色=正常, 黄色=预警, 红色=异常
3. 操作路径最短 — 核心操作(查看合同、更新进度)三步内完成
4. 移动端友好 — 至少核心看板能在手机上查看
5. 渐进式显示 — 先看概览，再点详情，再钻取明细
```

### 页面蓝图（按优先级排序）

```
P0 (MVP):
  /dashboard       — 综合看板 (5张卡片 + 3张图表)
  /contracts       — 合同台账 (搜索/筛选/排序)
  /contracts/:id   — 合同详情 (基本信息 + 阶段 + 付款 + 交付物)

P1:
  /finance         — 财务追踪 (开票vs回款对比 + 趋势)
  /stages          — 阶段进度看板
  /suppliers       — 供应商财务

P2:
  /reports         — 周报自动生成
  /alerts          — 风险预警
  /settings        — 系统配置
```

## 第7轮 · 实施路径层

### 阶段划分

```
阶段1: 数据基础 (本周)
  ├── 合同识别管线 (✅ DONE)
  ├── 数据清洗 (✅ 进行中)
  └── SQLite数据校验 (✅ DONE)

阶段2: 展示面板 (下周)
  ├── FastAPI后端 (✅ 已完成80%)
  ├── Vue3仪表盘布局
  ├── ECharts集成 (TOP10/趋势/分布)
  └── 合同列表 + 详情页

阶段3: 财务同步 (2周)
  ├── 飞书财务拉取脚本
  ├── 对比/导入/增量机制
  └── 财务追踪页面

阶段4: RuoYi融合 (3周+)
  ├── SQLite→MySQL迁移
  ├── RuoYi合同管理模块
  ├── 权限/工作流集成
  └── 数据同步cron

阶段5: 深化 (持续)
  ├── 阶段进度手动更新
  ├── 付款条件从合同补充
  ├── 风险预警机制
  └── 合同原文全文检索
```

### 关键决策点

```
决策1: SQLite还是MySQL作为主数据源？
  结论: SQLite是源数据，MySQL是RuoYi的管理镜像
  理由: SQLite单文件零运维，适合作为数据底座
        MySQL需要容器/连接池/RuoYi框架支持

决策2: 独立面板用什么前端框架？
  结论: Vue3 (与RuoYi一致的技术栈)
  理由: 组件可复用，风格统一，学习成本低

决策3: 图表库选型
  结论: ECharts (成熟稳定，中文文档好)
  备选: AntV G2 (更适合统计图表)

决策4: 合同详情中阶段/付款从何而来？
  结论: 当前从标准数据JSON读入DB
  优化: 后续从合同原文用LLM提取
```

## 第8轮 · 度量和验证层

### 成功标准

| 指标 | 当前值 | 目标值 | 测量方式 |
|------|--------|--------|----------|
| 合同金额完整度 | 95% (38/40) | 100% | DB中有金额的合同数 |
| 数据重复率 | ~20% (8对重复) | 0% | 唯一合同名/金额组合 |
| DB ↔ JSON 金额一致性 | 100% (已修复) | 100% | 交叉比对 |
| 阶段数据覆盖率 | 15% (6/40) | 50% | 有阶段数据的合同比例 |
| 付款数据覆盖率 | 17.5% (7/40) | 50% | 有付款计划的合同比例 |
| 仪表盘加载速度 | - | <2秒 | 首次渲染时间 |
| 飞书同步延迟 | 手动 | <24小时 | 飞书→DB时间差 |

### 验证清单

```
□ 合同总量 = 40个
□ 总金额偏差 < 5%
□ 无重复项目编号
□ 无零金额待补充 (2个已知例外)
□ 飞书财务数据可自动拉取
□ 仪表盘可展示核心KPI
□ 合同详情可查看阶段+付款
□ 系统可在5分钟内完成部署
□ 数据备份自动运行
```

### 回滚方案

```
如果阶段2(展示面板)出现问题:
  → 回退到 query_db.py 命令行查询
  → 所有数据在SQLite中不受影响
  
如果阶段3(财务同步)出现问题:
  → 回退到手动导入 (保持现有流程)
  → 财务数据仍在飞书表格中安全

如果阶段4(RuoYi融合)出现问题:
  → 独立面板不受影响
  → RuoYi侧回退到未定制状态
```
