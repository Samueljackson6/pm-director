# 审计整改需业务确认事项（2026-07-11）

## 目的

本文件记录审计整改中无法安全自动执行的事项。以下问题涉及主数据口径、合同源文件或认证体系，未经确认不得直接更新或删除生产数据。

## 1. 合同 canonical ID 规则

当前 `contract_id_mapping` 中存在 16 组同一 SGSC ID 对应多个 ZH/SYS ID。现有证据不足以确定：

- `ZH02-*` 是否统一作为内部主键；
- `SGSC*` 是否统一作为外部合同编号；
- `SYS-202607-*` 是否只是临时导入编号；
- 是否存在一个 SGSC 合法关联多个内部项目的业务情况。

### 禁止操作

- 禁止自动删除重复映射。
- 禁止仅凭 ID 前缀选择 canonical。
- 禁止在冲突未清理前增加唯一索引。

### 需要确认

1. 合同、项目、财务三类数据的事实主键分别是什么。
2. 一个 SGSC 编号是否允许关联多个项目。
3. `SYS-*` 数据应迁移、保留为别名还是删除。

## 2. 付款计划异常

实际 `payments` 字段为：

- `payment_id`
- `contract_id`
- `project_id`
- `payment_stage`
- `planned_amount`
- `actual_amount`
- `paid_amount`
- `payment_date`
- `planned_date`
- `paid_date`
- `status`

### 已确认异常

- `ZH02-202512056` 存在 `planned_amount=5123.0`，同时存在语义对应的 `0.5123` 记录，疑似万元/元误录和重复导入。
- `ZH02-202604026` 的合同金额与付款阶段合计不一致，且 `actual_amount` 使用了与 `planned_amount` 不同的单位口径。
- `ZH02-202408007` 同时存在付款计划不足和开票超合同金额问题。

### 禁止操作

- 禁止自动将金额除以或乘以 10000。
- 禁止仅凭同额或相似描述删除付款记录。
- 禁止自动修改合同金额以匹配付款计划。

### 需要确认

1. `planned_amount`、`actual_amount`、`paid_amount` 的持久化单位。
2. 付款计划总额是否必须等于合同含税金额。
3. 同额多笔付款是否允许存在。
4. 应以 PDF/DOCX 合同、财务 Excel 还是现有数据库哪个来源为修复真值。

## 3. 阶段时间与状态

79 条阶段中有 74 条缺少完整起止时间。部分时间区间被写入 `start_time`，前端通过文本解析进行补偿。

### 禁止操作

- 禁止根据阶段名称猜测日期。
- 禁止把单个日期自动扩展为时间区间。
- 禁止根据当前日期直接覆盖历史状态。

### 需要确认

1. 阶段是否强制同时拥有开始和结束日期。
2. 是否允许“合同签订后 30 日内”等相对时间文本。
3. `status` 是人工确认状态，还是应根据时间动态推导。

## 4. 认证改造

当前认证接口是开发 mock。真实认证改造前需要明确：

1. 使用 RuoYi Office 统一认证、独立 JWT，还是企业 SSO。
2. 用户、角色、菜单和数据权限的来源。
3. 生产环境是否允许保留本地应急管理员。
4. API 是否需要区分只读、编辑、财务和系统管理权限。

在认证方案确认前，本轮只完成 CORS 白名单和数据库 readiness，不擅自引入新的认证依赖或账户体系。
