## 二、已知问题

### 2.1 前端 Vben Admin 构建失败（关键问题）

**根因**: `ui-vben/` 目录是 Vben Admin 5.5.9 的源码快照，但部分文件缺失。

**具体缺失**:
1. `packages/@core/base/shared/src/cache/` — 整个目录缺失（已手动创建补全）
2. 多个 workspace 包未配置 `stub` 脚本，`unbuild --stub` 不会处理它们
3. 深层子路径导入如 `@vben/common-ui/es/loading` 不存在

**影响**: 不影响后端运行。前端开发可在 Windows 本地直接运行 `pnpm dev`。

**修复方案**: 从上游 Vben Admin 仓库 (`vbenjs/vue-vben-admin`) 同步缺失的 `packages/` 下各包的 `src/` 源码。

**已执行的操作**:
1. ✅ 从 GitHub 官方仓库克隆 v5.5.9 版本
2. ✅ 同步了 3 个缺失的 cache 模块源文件（`storage-manager.ts`, `types.ts`, `__tests__/*`）
3. ✅ 补全了缺失的静态资源占位文件（`statusBar.png`, `app-nav-bar-mp.png`, `iphone_backImg.png`, `menu_foot.png`, `menu_head.png`）

**深层根因**: 前端构建在 Vite 7 的 `commonjs--resolver` 插件中存在 Vite 别名与 jiti 模块解析的兼容性问题。
`node:fs/promises` 等 Node.js 子路径在通过别名映射时，会被错误地解析为
`stub.ts/promises`（将 .ts 文件当作目录）。且 `@vben/vite-config` 使用 jiti stub
（由 `unbuild --stub` 生成），而 jiti 的模块解析与 Vite 的别名系统存在冲突，
导致 `node:fs/promises` 等子路径在 CommonJS require 时无法正确解析。
此问题需等待 Vite 7.x 或 jiti 2.x 上游修复。

**临时解决方案**: 前端可以在 Windows 本地开发环境直接使用 `pnpm dev` 运行，
无需 Docker 容器化构建。

### 3.2 本地 Docker vs 生产对比

| 维度 | 生产 (236) | 本地 Docker | 一致? |
|------|-----------|-------------|-------|
| Python 版本 | 3.11 (系统) | 3.11-slim (容器) | ✅ |
| 后端命令 | `uvicorn backend.main:app --host 0.0.0.0 --port 8800` | 同左 | ✅ |
| 数据库 | 本地 SQLite 文件 | Volume 挂载 SQLite 文件 | ✅ |
| 依赖 | `requirements.txt` | 同左 (Dockerfile 安装) | ✅ |
| 前端服务 | python http.server | nginx:alpine | ⚠️ 不同 |
| Node.js | 宿主机 v22 | 构建容器 node:20 | ⚠️ v22 vs v20 |
| 操作系统 | Ubuntu 24.04 主机 | Debian/Alpine 容器 | ⚠️ 不同 |

### 3.3 一致性评估

**后端**: ✅ 基本一致。Python 版本、依赖、启动命令、数据库均相同。

**前端**: ⚠️ 有差异但可接受。Docker 使用 nginx 是生产环境的改进方向，
生产环境未来也会迁移到容器化部署（见 MEMORY.md 的部署架构变更计划）。
