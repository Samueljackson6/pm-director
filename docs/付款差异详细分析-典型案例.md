## 汇总发现

### 问题类型统计

| 问题类型 | 出现次数 | 影响 |
|---------|---------|------|
| 多阶段合并付款被拆分 | 5+ | 导致金额重复计算 |
| 阶段与付款关联错误 | 10+ | 付款阶段描述不准确 |
| OCR识别金额缺失 | 多处 | 部分金额未提取 |
| 重复付款记录 | 2+ | 直接导致金额超出 |

### 根因分析

1. **数据提取脚本问题**:
   - 无法正确处理"5.1-5.2合并付款"这种多阶段合并的情况
   - 把一个合并付款错误拆分为多条记录

2. **阶段与付款关联问题**:
   - payment_stage字段的文本描述不一致
   - 有的是"研究阶段5.1"，有的是"条款5.1约定内容"

3. **去重逻辑缺失**:
   - 相同金额+相同条件的付款记录没有去重

## 修复建议

### 立即行动

1. **删除明显的重复付款**:
```sql
-- ZH02-202408007 重复的18.03万元
DELETE FROM payments 
WHERE contract_id = 'ZH02-202408007' 
  AND planned_amount = 18.03 
  AND rowid NOT IN (SELECT MIN(rowid) FROM payments WHERE contract_id = 'ZH02-202408007' AND planned_amount = 18.03);
```

2. **删除明显多余的付款记录**:
```sql
-- ZH02-202507011 多余的10.0万和9.0万
DELETE FROM payments WHERE contract_id = 'ZH02-202507011' AND planned_amount IN (10.0, 9.0);
```

### 长期改进

1. **改进付款阶段解析逻辑**:
   - 识别"5.1-5.2"、"阶段5.1+阶段5.2"等合并描述
   - 不要拆分为多条记录

2. **添加数据校验**:
   - 付款总额应在合同金额±误差范围内
   - 自动检测超出合同金额的异常情况

3. **改进数据清洗脚本**:
   - 添加基于金额+条件+阶段的去重逻辑
   - 标记需要人工确认的异常数据
