# 阶段 3 任务细化与 Demo 评审方案

更新日期：2026-07-15
当前状态：🔄 进行中，已完成可交互 Demo，等待用户评审
实施边界：本轮完成方案细化、Demo 和后端适配审查，不提前改造生产路由与业务数据

## 1. 阶段目标

阶段 3 的目标是把已经确认的信息架构和 UI 规范转换为可评审、可实施、可验收的产品方案：

1. 让用户从任务域进入业务，而不是在多套旧模块中寻找功能。
2. 让合同、项目、发票、回款、供应商发票和付款形成连续上下文。
3. 用“履约链”建立 PM Director 的视觉与业务识别度。
4. 在美观、科技感和信息密度之间保持平衡。
5. 在生产改造前，通过静态高保真 Demo 锁定交互和视觉方向。

## 4. 视觉方向

### 4.1 设计主题

主题名称：安静的未来指挥台

- 深海蓝用于全局导航和沉浸式深色主题。
- 云雾白用于高密度业务内容和长时间阅读。
- 电光青蓝只用于履约链、当前状态、数据连线和焦点反馈。
- 红、橙、绿只表达风险、待办和完成，不作为装饰色。
- 等宽字体只用于编号、金额、日期和数据时点。

### 4.2 识别元素

“履约链”是唯一强视觉中心：

```text
合同签订 → 项目立项 → 阶段/交付物 → 发票 → 回款/付款
```

桌面端采用横向轨道，移动端采用纵向时间线。已完成节点使用稳态蓝，当前节点使用电光青，未开始节点降低对比度。

### 4.3 动效边界

| 场景 | 时长 | 规则 |
|------|------|------|
| 页面进入 | 160–180ms | 淡入并上移 4px |
| 按钮/表格反馈 | 120–140ms | 颜色、边框或 1–2px 位移 |
| 履约链首次点亮 | 180–480ms | 只播放一次，不循环 |
| 导航抽屉 | 180ms | 进入和退出保持一致 |
| reduced-motion | 0ms | 关闭位移与非必要动画 |

禁止持续背景粒子、自动轮播、循环数字、全卡片发光和大面积霓虹渐变。

## 5. 页面与 URL 状态契约

### 5.1 驾驶舱

```text
/dashboard/overview/index?view=all|management|projects|finance
```

刷新后必须保持视图；行动项只由页面容器执行一次导航。

### 5.2 列表

```text
/delivery/contracts?q=管廊&status=active&page=1&sort=next_action
/receivables/invoices?payment_status=overdue
```

筛选、页码和排序进入 URL；目标列表必须初始化并监听 Query，而不是只改变地址栏。

### 5.3 详情与返回

```text
/delivery/contracts/detail?id=18&from=/delivery/contracts?...完整列表状态
/delivery/projects/detail?id=42&from=/delivery/contracts/detail?...完整合同来源
```

`from` 只接受内部规范路由。直接打开详情时使用规范列表作为回退；任何旧路径只做保留 Query 的单向重定向。

## 7. 后端与路由适配审查

### 7.1 P0：生产实现前必须完成

1. 注册 `/delivery/*`、`/receivables/*` 和 `/supplier-fulfillment/*` 规范路由，并为项目执行补正式菜单。
2. 移除供应商详情自重定向，避免 `/suppliers/detail` 循环。
3. 修正供应商详情新增付款请求：真实接口为 `/api/suppliers/payments`。
4. 统一并校验 `from` 回跳契约，只允许内部路径。
5. 旧项目路由不得再重定向到驾驶舱或合同详情。

### 7.2 P1：首个纵向切片完成

1. 提醒组件只上报动作，不在子组件和父组件各跳转一次。
2. 合同、项目、客户发票和供应商列表读取并监听 Query。
3. 扩展合同待交付、付款逾期、未开票等筛选。
4. 新增轻量任务明细接口，返回目标对象、原因、截止时间、责任人和状态。
5. 统一金额单位响应契约并补齐 TypeScript 类型。

### 7.3 P2：根据性能数据决定

- 评估是否把项目详情的多次并行请求聚合为单一详情接口。
- 只在真实性能数据证明有收益时实施，避免提前扩大后端改造。

## 8. 实施批次

### 批次 A：路由与壳层

- 修复项目和供应商 P0 路由问题。

- 切换单侧栏并关闭全局页面标签栏。

### 批次 B：合同纵向切片

- 将合同台账、合同详情和项目详情映射到 Demo 方案。
- 复用现有合同详情数据在前端组装履约链。
- 验证来源返回和列表状态恢复。

### 批次 C：客户应收

- 分离客户发票与供应商入站发票。
- 统一发票、回款和匹配状态。
- 完成驾驶舱逾期任务钻取。

### 批次 D：供应商履约

- 合并供应商档案、合同、发票和付款上下文。
- 修正付款接口并完成逐级返回。

### 批次 E：全量验收

- 路由与任务流 E2E。
- 1440px、1024px、390px 响应式检查。
- WCAG AA、键盘、reduced-motion 和性能预算。
- 对比运行时构建版本，确认部署内容与源码一致。

## 10. 用户评审清单

请重点确认：

1. 四个一级任务域是否符合日常工作语言。
2. 驾驶舱四视图是否保留，是否需要调整默认视图。
3. 深海蓝、云雾白和电光青蓝是否符合期望的科技时尚感。
4. “履约链”是否适合作为详情页核心识别元素。
5. KPI、趋势、行动队列和表格的信息密度是否合适。
6. 合同→项目→原路返回是否符合使用习惯。
7. 客户应收和供应商履约是否需要调整菜单名称。
8. 明暗主题是否都需要进入生产版本。

评审结论确认后，阶段 3 才进入生产实现；评审前不扩大到新的工作台或额外业务模块。
