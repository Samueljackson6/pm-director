# 阶段 3 信息架构与规范路由

> 日期：2026-07-15
>
> 状态：设计基线 v0.1，待首击测试与纵向切片验证
> 适用范围：`ui-vben/apps/web-antd` 与 `backend/routers/auth.py`

## 1. 设计结论

PM Director 的全局信息架构改为四个任务域，只保留左侧栏作为全局导航：

```text
驾驶舱
合同履约
  ├─ 合同台账
  └─ 项目执行
客户应收
  ├─ 发票台账
  └─ 回款记录
供应商履约
  ├─ 供应商档案
  ├─ 供应商发票
  └─ 付款记录
```

详情页不在菜单中显示。
用户必须从列表、驾驶舱行动项或关联对象进入详情，并能可靠返回原工作上下文。

这套结构按用户任务组织，不按数据库表或开发模块组织：

- 驾驶舱负责判断当前最需要处理什么。
- 合同履约负责从合同承诺跟进到项目与交付。
- 客户应收负责从开票跟进到回款。
- 供应商履约负责从供应商档案跟进到发票和付款。

## 2. 当前事实与问题

当前源码和本地 `18080` 菜单 API 一致，提供五个一级菜单：概览、合同管理、客户财务、供应商管理、供应商财务。
项目执行页面不在后端菜单中。

当前 `preferences.ts` 使用 `mixed-nav`，运行截图同时出现顶部菜单、左侧菜单和页面标签栏。
同一层级的三个导航入口竞争注意力，也使隐藏页面和旧页面更难识别。

当前项目列表和详情源码仍存在，但运行路由把 `/projects` 重定向到驾驶舱，把 `/projects/detail` 重定向到合同详情。
当前供应商详情静态路由还存在指向自身的重定向，运行时会导致递归和栈溢出。
驾驶舱行动项由父子组件分别执行导航，列表又没有消费对应筛选 Query，因此一次行动可能发生双跳并丢失筛选语义。

当前驾驶舱已确认采用 `/dashboard/overview/index` 单一路由和 `?view=all|management|projects|finance` 四视图。
阶段 3 保留这项已确认决策，不扩张为新的多路由工作台。

## 3. 全局导航规则

1. 桌面端使用 Vben `sidebar-nav` 布局。
2. 顶栏只保留搜索、刷新、主题、用户等工具，不承载一级业务菜单。
3. 关闭全局页面标签栏，避免它继续充当第三套导航。
4. 启用面包屑，用于表达位置，不用于替代业务操作。
5. 页面内 Tab 只表达同一对象或同一路由下的视图切换。
6. 驾驶舱四视图是页面内局部导航，必须写入 URL Query。
7. 详情页隐藏于菜单，并通过 `activePath` 保持父级菜单高亮。
8. 一级菜单最多四项，层级最多三级，常用任务最多两次点击到达列表。
9. 菜单名称使用用户能识别的业务词，不使用“综合”“全域”“中心”等空泛词。

## 4. 目标菜单与路由树

```text
/dashboard                         驾驶舱
  └─ overview/index                经营驾驶舱（菜单内折叠为一级入口）

/delivery                          合同履约
  ├─ contracts                     合同台账
  ├─ contracts/detail              合同详情（隐藏）
  ├─ projects                      项目执行
  └─ projects/detail               项目详情（隐藏）

/receivables                       客户应收
  ├─ invoices                      发票台账
  ├─ invoices/detail               发票详情（隐藏）
  ├─ receipts                      回款记录
  └─ receipts/detail               回款详情（隐藏）

/supplier-fulfillment              供应商履约
  ├─ suppliers                     供应商档案
  ├─ suppliers/detail              供应商详情（隐藏）
  ├─ invoices                      供应商发票
  ├─ invoices/detail               供应商发票详情（隐藏）
  ├─ payments                      付款记录
  └─ payments/detail               付款详情（隐藏）
```

父级路径使用绝对路径，所有子级路径使用相对路径。
组件仍从 `views/` 按 Vben 后端路由模式解析，不在前端业务模块中新增平行路由。

## 6. 页面关系规则

### 6.1 列表

- 筛选、页码、排序写入 URL Query。
- 对象名称或编号使用显式链接进入详情。
- 行双击可以作为增强，但不能是唯一入口。
- 驾驶舱行动项进入列表时携带可见筛选条件，例如 `status=overdue`。
- 空结果必须说明当前筛选条件，并提供“清除筛选”。

### 6.2 详情

- 页首显示“返回来源”，不得只显示固定的“返回列表”。
- 标题使用对象名称，编号作为辅助标识。
- 面包屑表达信息层级，返回按钮表达操作历史，两者职责不同。
- 关联对象使用真实对象名和类型，例如“项目：电力隧道智能监测”，不使用泛化的“查看详情”。
- 直接打开详情时，没有可信来源则回到该对象的规范列表。

### 6.3 关联对象

- 合同可进入项目、客户发票和回款。
- 项目可返回关联合同。
- 客户发票可进入关联合同和回款。
- 供应商可进入供应商发票和付款。
- 关联跳转必须把当前详情的完整规范路径作为安全内部来源。

## 7. 角色与入口

阶段 3 不新增尚不存在的角色权限模型。
所有当前用户先使用同一套菜单树，驾驶舱默认视图根据工作需要切换：

| 使用者 | 推荐默认视图 | 首要任务 |
|---|---|---|
| 项目总监 | `all` 或 `management` | 判断整体风险和经营状态 |
| 项目经理 | `projects` | 跟进阶段、交付和负责人缺失 |
| 财务人员 | `finance` | 跟进开票、回款和异常款项 |
| 供应商协同人员 | 供应商履约菜单 | 跟进发票、付款和供应商资料 |

后续只有在后端已经提供真实角色和权限码时，才允许配置角色默认首页或隐藏菜单。

## 8. 迁移与切换顺序

1. 先在后端菜单中建立目标父级和规范路径，保持现有组件不变。
2. 删除供应商详情自重定向和驾驶舱行动项双跳转，保证迁移入口本身可验证。
3. 为所有旧路径建立 `Legacy*` 单向重定向，透传 `id`、筛选和来源参数。
4. 更新页面内部所有命名路由和硬编码路径。
5. 让目标列表实际读取并展示驾驶舱传入的筛选 Query。
6. 实现统一来源与返回契约。
7. 先交付“合同台账 → 合同详情 → 项目详情 → 返回”的纵向切片。
8. 通过路由 E2E、可访问性和首击测试后，再迁移客户应收和供应商履约。
9. 旧路径在退役日前只允许重定向；退役日后删除旧页面与重复路由。

## 9. 验收标准

- 全局只存在一套业务导航。
- 一级菜单为四个任务域，项目执行有正式入口。
- 每个业务对象只有一个规范名称、路径和组件。
- 详情页不在菜单中显示，但父级菜单高亮正确。
- 所有旧路径只向新路径重定向，没有反向引用。
- 主要任务不经过 404、隐藏旧页或兼容页面。
- 列表筛选、页码和排序可在刷新后恢复。
- 详情返回恢复来源页面及其工作上下文。
- 运行菜单 API、磁盘 `auth.py` 和已部署构建的路由摘要一致。

## 10. 设计边界

- 本文不更换 Vue、Vben、Ant Design Vue 或 FastAPI。
- 本文不新增业务模块和数据字段。
- 本文不把不确定数据自动改成确定状态。
- 本文不授权直接修改生产环境。
- 规范路由注册表需在纵向切片实现前完成一次产品确认。
