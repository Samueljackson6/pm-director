# 金额单位统一策略决策

> 日期：2026-07-20  
> 状态：**已确认**  
> 关联 Issue：[#48 P0-1](https://github.com/Samueljackson6/pm-director/issues/48)

---

## 决策

**数据库保持万元/元混合现状，API 层统一输出为元。**

---

## 理由

1. **避免破坏历史数据**：直接修改数据库可能导致财务快照、历史报表不一致
2. **最小化代码变更**：只需在 API 输出层做单位转换，不改动数据写入逻辑
3. **前端统一接收元**：前端展示时自行转换为万元（÷10000），符合用户阅读习惯

---

## 各表单位现状

| 表/字段 | 数据库单位 | API 输出单位 | 转换逻辑 |
|---------|-----------|-------------|---------|
| `contracts.contract_amount` | 万元 | 元 | ×10000 |
| `projects.total_contract_amount` | 万元 | 元 | ×10000 |
| `payments.planned_amount` | 万元 | 元 | ×10000 |
| `payments.paid_amount` | 万元 | 元 | ×10000 |
| `finance_records.*_total` | 万元 | 元 | ×10000 |
| `current_finance_view.*` | 万元 | 元 | ×10000 |
| `supplier_contracts.contract_amount` | 万元 | 元 | ×10000 |
| `invoices.amount` | 元 | 元 | 不变 |
| `receipts.amount` | 元 | 元 | 不变 |
| `master_table.含税金额` | 元 | 元 | 不变 |

---

## 前端处理规则

1. 所有金额字段在展示时统一除以 10000 转为万元显示
2. 输入框接受元或万元，由用户选择，后端统一存储原单位
3. 金额对比时自动换算到同一单位

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 前端忘记 ÷10000 | 显示 10000 倍数值 | 建立金额格式化工具函数，统一调用 |
| 部分 API 遗漏转换 | 数据不一致 | 编写单元测试覆盖所有金额字段 |
| 历史快照表单位混乱 | 审计困难 | 在 API 响应中标注数据来源和单位 |
| 供应商付款单位未确认 | 可能显示错误 | 先分析 supplier_payments 数据再决定 |
