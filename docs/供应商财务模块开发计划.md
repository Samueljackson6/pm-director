### 1.3 已有后端 API

**供应商相关 API** (`backend/routers/suppliers.py`):
- `GET /api/suppliers` - 供应商列表
- `GET /api/suppliers/{id}` - 供应商详情（包含发票和付款）
- `POST /api/suppliers/payments` - **新增供应商付款记录（已实现）**

**发票相关 API** (`backend/routers/invoices.py`):
- `GET /api/invoices?direction=inbound` - 可获取供应商发票（direction=inbound）
- `GET /api/invoices/{id}` - 发票详情
- `POST /api/invoices` - 新增发票
- `PUT /api/invoices/{id}` - 更新发票
- `DELETE /api/invoices/{id}` - 删除发票

### 1.4 数据库表结构

**suppliers 表** - 供应商主表:
- supplier_id, supplier_name, short_name, credit_code
- total_contract_amount, total_invoiced, total_paid, total_unpaid

**supplier_contracts 表** - 供应商合同关联:
- project_id, supplier_id, contract_amount
- invoice_total, payment_total, unpaid

**supplier_payments 表** - 供应商付款:
- supplier_id, project_id, payment_date, amount, payment_method, status

**invoices 表** - 发票（通过 direction='inbound' 区分供应商发票）:
- direction='outbound' → 客户发票（我们给客户开）
- direction='inbound' → 供应商发票（供应商给我们开）

## 四、技术实现建议

### 4.1 复用客户财务模块代码

供应商财务与客户财务业务逻辑高度相似，建议：

1. **复制客户财务模块文件**作为基础
2. **调整字段映射**:
   - `客户发票` → `供应商发票`
   - `客户回款` → `供应商付款`
   - `客户` → `供应商`
   - `应收` → `应付`
   - `receipt` → `payment`

### 4.2 数据模型复用

```typescript
// 供应商发票接口（复用 InvoiceItem，direction='inbound'）
interface SupplierInvoice {
  invoice_id: number
  supplier_id: string
  supplier_name: string
  project_id: string
  invoice_no: string
  invoice_date: string
  amount: number
  tax_rate: number
  status: '待付款' | '部分付款' | '已付款'
}

// 供应商付款接口
interface SupplierPayment {
  payment_id: number
  supplier_id: string
  supplier_name: string
  project_id: string
  payment_date: string
  amount: number
  payment_method: string
  status: string
}
```

### 4.3 API 命名规范

建议创建独立的 API 路由：
- `/api/supplier-invoices` - 更清晰地区分客户发票和供应商发票
- `/api/supplier-payments` - 更清晰地区分客户回款和供应商付款
