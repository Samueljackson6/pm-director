# 合同详情页整体改造方案

> 日期：2026-07-11 | 基于：审计报告、真实数据库复算、Playwright 实地渲染、提取/清洗脚本审查
> 范围：数据质量 → 提取/清洗规则 → 前端展示，三层整体改造

---

## 〇、核心结论（一句话）

前端"显示错"只是表象，**真正的根因在「回款数据根本没进 payments 表」+「同一合同被 SGSC/ZH 双 ID 各存一份导致 OR 查询翻倍」+「金额单位启发式脆弱」**。三层必须一起改，但**数据库清洗属于业务确认红线，必须你授权后才能执行**。

## 一、数据质量现状（已量化，只读复算）

| 维度 | 现状 | 影响 |
|------|------|------|
| 合同主数据 | 42 条（ZH36 / SGSC6 / SYS0）；`contract_id_mapping` 49 行（含过期重复）；**14 条 ZH 合同缺映射** | 双 ID 匹配放大 |
| 孤儿引用 | `SYS-202607-001` 在 payments 中被引用，但 **不在 contracts 表** | 该合同付款无归属 |
| 付款单位/重复 | ZH02-202512056 同时有 `SGSCYA00GYJS2500409-付款1` 与 `ZH02-202512056-P1` 两套付款（同额）→ OR 查询翻倍；contract_amount 疑为 5123.0（单位错） | 付款计划膨胀 302× |

| 回款真实来源 | `invoices` 中 `invoice_type='客户回款'`：34 行 / SUM=659.67 万 ✅；`payments.paid_amount` SUM=**200.0（≈0）** ❌ | **回款率算出来恒为 0** |

**致命发现（已 100% 确认）**：
- 回款真实存在于 `invoices`（类型=`客户回款`），**不在 `payments.paid_amount`**。
- 前端 `detail.vue` 用 `receiptRate = payments.paid / payments.planned` → 永远 0。
- 前端 KPI「已开票」把**所有 invoices（含客户回款）一起求和** → 把回款也算进开票 → 与 finance 快照矛盾。

## 二、提取/清洗规则评估（scripts/）

| 脚本 | 问题 | 优化建议 |
|------|------|---------|
| `import_to_db.py:115` | `if amt > 1000: amt = amt/10000` **脆弱启发式**：≤1000 的"元"值不除 → 单位错乱；>1000 的"万"值被错除 | 改为**显式单位制**：标准数据 JSON 携带单位字段；导入时统一换算，禁止靠数值大小猜 |

| `batch_extract_docx.py` | `infer_project_type` 关键词猜测（科研/服务/物资）不可靠 | 低优先级；保留但标注"AI推测" |

## 四、整体改造方案（三层）

### L1 展示层修复 —— 安全，纯代码，不碰数据 【可立即执行】

**后端 `backend/routers/contracts.py`（get_contract）**
- KPI 字段统一从 `finance_records` / `current_finance_view` 取（与 dashboard 同源），不再用原始 invoices/payments 求和。
- `payments` 查询去重：经 `contract_id_mapping` 取 canonical id 后**只匹配 canonical 一套**，杜绝 SGSC+ZH 双份。
- 发票明细按 `invoice_type` 拆分：开票=`客户开票`、回款=`客户回款`；状态映射 `已开/已回款`。

**前端 `detail.vue`**
- KPI 行改用 `overview.finance` 字段（已回款=`payment_total`，回款率=`payment_total/contract_total`）。
- 删除 `receiptRate = payments.paid/planned` 计算。
- 阶段表：start/end 为空显示「数据待补」；移除与付款的下标配对，二者各自独立展示。
- 发票明细按类型分栏，状态标签用映射表。

**效果**：F1/F2/F3/F4/F5/F6 全部消除；详情页与 dashboard 数字一致。

### L2 提取/清洗规则优化 —— 安全，纯代码 【可立即执行】

- `import_to_db.py`：金额改为显式单位制 + 付款按 (canonical, 金额, 节点) 去重 + 合同改名级联更新付款。
- 新增 `scripts/validate_import.py`：导入后断言——合同额量级合理（>1000 万告警）、无孤儿引用、mapping 覆盖完整。
- 文档化「ZH02=canonical 主键」铁律，写入 `AGENTS.md`/知识库。

**效果**：新合同不再产生双 ID / 单位错；防止问题复发。

### L3 数据库数据清洗 —— 触及数据，需业务确认红线 【准备 SQL，待你授权】

> 依据项目铁律（飞书增量维护、合同审计"需业务确认项"）：**不擅自删除/除以 10000/猜时间**。以下 SQL 已写好但**不执行**，等你确认 canonical ID 与口径后运行。

1. **去重付款**：对同时有 SGSC/ZH 付款的合同，保留 canonical（ZH）一套，删除别名重复行。
2. **修正单位**：ZH02-202512056 合同额/付款额按确认的元/万口径校正（需你拍板 5123 究竟是元还是万）。
3. **补映射**：为 14 条缺映射 ZH 合同从 `cache/contracts/*-标准数据.json` 补 `contract_id_mapping`。
4. **处理孤儿**：`SYS-202607-001` 付款 → 要么补建合同，要么重定向到其 canonical ID。
5. **刷新快照**：`finance_records` 按实时发票重算（解决 94.85 万 vs 254.77 万过期）。

## 五、优先级与执行顺序

```
P0  L1 后端 get_contract 统一 finance 源 + 付款去重      ┐
P0  L1 前端 detail.vue KPI/阶段/发票修复                 ├─ 立即可做（代码，可逆）
P0  L2 import_to_db 单位制 + 去重 + 改名级联             ┘
P1  L3 数据库清洗 SQL（去重/单位/映射/孤儿/快照）        ── 需你授权后执行（数据，红线）
```

**架构建议（治本）**：确立 `finance_records` 为合同财务**唯一可信源**（已去重、按规范修正），前端/后端所有金额类展示一律读它；原始 `invoices`/`payments` 仅用于"明细列表"展示，不再参与汇总计算。这样能一劳永逸消除"双源矛盾"和"回款率=0"。
