## 二、逐项详细说明

### C1 — KPI 行与财务汇总卡片数值矛盾（影响全部合同）

#### 现象

以 ZH02-202601011 为例（截图 `cd-ZH02-202601011.png` 可见）：

| 指标 | KPI 行（顶部4张卡） | 财务汇总卡片（右侧） | 是否一致？ |
|------|-------------------|---------------------|-----------|
| 已开票 | **254.77 万** | **94.85 万** | ❌ 差 2.7× |
| 已回款 | **0.00 万** | **52.85 万** | ❌ 完全不同 |
| 未回款 | **496.60 万** | **443.75 万** | ❌ 不同基数 |

用户在同一个页面上看到两套互相矛盾的财务数字。

#### 根因

**KPI 行**从 `detail.invoices` 列表和 `detail.payments` 列表计算：
- `invoiceTotal` = Σ(invoices.amount) ÷ 10000 → **汇总了所有发票（含「客户回款」类型）**
- 已回款 = `paymentsTotal.paid` → **来自 payments 表的 paid_amount 字段**

**财务汇总卡片**从 `detail.finance` 对象取值（后端 `finance_records` 或 `current_finance_view` 快照）：
- invoice_total → **仅统计「开票」类发票**
- payment_total → **来自财务快照的真实回款记录**

两个来源口径完全不同，但都标注为"已开票"/"已回款"。

#### 前端代码位置

```
detail.vue
  L1388-1395  invoiceTotal computed: sum(invoices.amount)/10000  ← 含回款发票
  L1405-1408  receiptRate computed: paid/planned               ← 付款进度非回款率
  L1411-1413  unreceivedAmount: planned - paid                  ← 付款未付非未回款

vs

  L228-262    财务汇总卡片: finance.invoice_total / finance.payment_total ← 正确口径
```

#### 修复建议

**方案 A（推荐）：统一 KPI 行使用 finance 快照源**
- KPI 的已开票 = `finance.invoice_total`
- KPI 的已回款 = `finance.payment_total`
- KPI 的未回款 = `finance.payment_unreceived`
- 开票率 = invoice_total / contract_amount × 100%
- 回款率 = payment_total / contract_amount × 100%
- 保留发票/付款明细作为补充展示（去掉顶部 KPI 中的重复计算）

**方案 B：保留双源但明确标签区分**
- KPI 行改名为「发票合计」「付款合计」等精确名称
- 财务汇总保持「已开票」「已回款」

### C2 — ZH02-202512056 付款计划总额膨胀 302 倍（P0 内容错误）

#### 现象

该合同合同额 = **17.08 万元**。

页面付款进度区显示：
- 付款阶段：**4 个**
- 计划总额：**5156.64 万元** ← ❌❌❌ 错误！应为 ~17.08 万
- 已付金额：0.00 万元
- 付款进度：0.0%

#### 根因

该合同的 `payments` 表存在 **双重 ID 记录 + 单位混用**：

| payment_id | contract_id | project_id | planned_amount | 说明 |
|-----------|-------------|------------|----------------|------|
| SGSCYA00GYJS2500409-付款1 | None | ZH02-202512056 | 16.5643 | ✅ 万元（正确） |
| SGSCYA00GYJS2500409-付款2 | None | ZH02-202512056 | **5123.0** | ❌ 元（应为 0.5123万） |
| ZH02-202512056-P1 | ZH02-202512056 | ZH02-202512056 | 16.5643 | ✅ 万元（正确，与上面重复） |
| ZH02-202512056-P2 | ZH02-202512056 | ZH02-202512056 | 0.5123 | ✅ 万元（正确，与上面重复） |

- `SGSCYA00GYJS2500409-付款2` 的 `planned_amount=5123.0` 是**元**（0.5123万的元值），前端当万元展示 → 显示 5123.00 万
- 同一付款在 SGSC ID 下存了一次（元值）、在 ZH ID 下又存了一次（万元值）→ **重复记录**
- 后端查询用了 4-way OR (`contract_id=mapped OR project_id=mapped OR contract_id=orig OR project_id=orig`)，把两个 ID 系统下的记录全部拉出

#### 前端代码位置

```
backend/routers/contracts.py L143-146  payments 查询（4-way OR 过宽）
detail.vue L1373-1379           paymentsTotal 汇总（直接求和，无单位校验）
```

#### 修复建议

1. **数据层**：清洗 payments 表，统一为万元；去重（保留 ZH id 版本，删除 SGSC 版本）
2. **后端**：收紧查询条件，优先 `contract_id=zh_mapped_id`，避免 4-way OR 拉入无关/重复行
3. **前端**：增加防御性校验——若 payments SUM >> contract_amount × 安全倍数，标记异常

### L1 — 回款率永远是 0%（P1 逻辑关联错误）

#### 现象

所有被测合同的 KPI 回款率均显示 **0.0%**，已回款均为 **0.00 万**。

但财务汇总卡片显示真实回款（如 ZH02-202601011：52.85 万，ZH02-202408007：67.44 万）。

#### 根因

`receiptRate` 计算公式：

```javascript
// detail.vue L1405-1408
const receiptRate = computed(() => {
  if (paymentsTotal.value.planned === 0) return '0.0'
  return ((paymentsTotal.value.paid / paymentsTotal.value.planned) * 100).toFixed(1)
})
```

分子 `paymentsTotal.paid` 来自 `payments.paid_amount` —— 该字段在 DB 中**全部为 0 或 null**（付款表只记录计划金额和完成标记，未录入实际到账金额）。

实际回款信息存储在：
- `invoices` 表中 `type='客户回款'` 且 `status='已回款'` 的 amount
- `finance_records` / `current_finance_view` 的 `payment_total` 字段

页面把"回款"这个业务概念绑定到了错误的字段上。

#### 修复建议

- KPI "已回款"应取 `finance.payment_total`（或 Σ(客户回款发票.amount)/10000）
- "回款率" = payment_total / contract_amount × 100%
- 若需保留付款进度视图，单独命名为"付款执行率"并明确基于 payments 表

### L2 — 阶段↔付款映射是编造的（P1 逻辑关联）

#### 现象

阶段详情表的"阶段金额"列显示了金额数字（如 ZH02-202601011 阶段1 显示 ¥496.60 万）。这些数字看起来合理但**不可信**。

#### 根因

```javascript
// detail.vue L1251-1286  stagePaymentMap
// 策略：将付款按排序顺序一对一分配给阶段
// 付款数 <= 阶段数 → 一对一
// 付款数 > 阶段数 → 最后一对多
```

没有任何 FK 关联阶段和付款。纯靠数组索引分配：
- 5 阶段 1 付款 → 付款全挂到阶段 1，其余阶段显示 "-"
- 3 付款 5 阶段 → 前 3 阶段各挂一个，后 2 阶段 "-"

如果付款顺序和阶段顺序不是一一对应的（实际情况通常如此），则每个阶段显示的金额都是错的。

#### 修复建议

- 短期：移除"阶段金额"列或标注"⚠️ 仅供参考（自动匹配，可能不准确）"
- 中期：在 DB 中建立 `payment_stage_link` 表（payment_id ↔ stage_id FK）
- 或在 payments 表中加 `stage_number` 外键字段

### L3 — 发票↔付款启发式匹配（P2 逻辑关联）

#### 现象

每条付款下方的「已开票/未开票」「✅ 已回款 / ⚠ 未回款」标签是基于金额近似匹配猜测的。

#### 根因

```javascript
// detail.vue L1347-1358
Math.abs(inv.amount - plannedAmount * 10000) < 1000  // 误差<1000元视为匹配
```

- 发票 amount 是元，付款 planned 是万元 → *10000 比较
- 误差阈值 1000 元对于大额发票太宽松，对小额发票太紧
- 多个发票金额接近时，总是选最早的那个
- 如果付款金额本身就有单位错误（C2），匹配必然失败

#### 修复建议

- 短期：移除付款卡片中的发票匹配标签（当前准确率极低）
- 中期：在 invoices 表中加 `payment_id` 外键，或在 payments 表中加 `matched_invoice_id`

### D1 — 发票状态标签永远显示「待回款」（P2 显示缺陷）

#### 现象

关联发票列表中每张发票的状态标签均为橙色「待回款」，即使该发票 type='客户回款' 且 status='已回款' 也如此。

#### 根因

```javascript
// detail.vue L627-631
<a-tag :color="inv.status === 'paid' ? 'green' : 'orange'" size="small">
  {{ inv.status === 'paid' ? '已回款' : '待回款' }}
</a-tag>
```

DB 中发票 status 取值为 `'已开'` / `'已回款'`（中文），而前端判断的是英文 `'paid'` → **永远不命中**。

#### 修复

```javascript
// 改为
const invPaid = inv.status === '已回款' || inv.status === 'paid'
:color="invPaid ? 'green' : 'orange'"
{{ invPaid ? '已回款' : '待回款' }}
```

### D2 — 付款 completed ≠ paid（P3 显示缺陷）

#### 现象

部分付款 DB status='completed'（表示该付款阶段已完成审批），但页面因为 `isPaid = p.status === 'paid'` 将其标为橙色「待支付」。

实际上 DB 存在三套状态值：`pending`, `completed`, `paid`，语义分别为：
- `pending`：待处理
- `completed`：已完成（可能是审批通过但尚未打款确认）
- `paid`：已支付（款项到账）

页面只认 `paid`，忽略了 `completed`。

#### 修复

```javascript
const isPaid = p.status === 'paid' || p.status === 'completed'
```
