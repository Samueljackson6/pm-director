# pm-director 约束与执行规范

> 版本: 1.0 | 更新: 2026-07-02 | **强制执行**

---

## 一、总则

所有开发和维护工作必须遵循本文档定义的约束和规范。任何违反可能导致：

- 代码无法部署
- 服务不可用
- 数据丢失
- 需要回滚

---

## 二、开发约束

### 2.1 修改范围约束

| 规则 | 说明 | 违规后果 |
|------|------|---------|
| **只改必要文件** | 所有改动必须和当前任务直接相关 | Code Review 拒绝 |
| **保持代码风格** | 继承项目原有风格，不要混用 | CI 警告 |
| **禁止顺手重构** | 不相关的代码不要动 | 回滚 |
| **禁止新依赖** | 除非充分理由并记录 | CI 拒绝 |

### 2.2 前端约束

| 规则 | 说明 | 检查方式 |
|------|------|---------|
| 必须使用 `requestClient` | 禁用 `baseRequestClient` | 人工审查 |
| 必须使用 `useVbenVxeGrid` | 禁用原生 `a-table` | 人工审查 |
| 子路由 `path: ''` | 禁用 `path: 'index'` | grep 检查 |
| 遵循 Vben 规范 | 见 `docs/Vben开发规范.md` | 人工审查 |

### 2.3 后端约束

| 规则 | 说明 |
|------|------|
| FastAPI + Pydantic v2 | 统一框架 |
| CORS `allow_origins=['*']` | 开发环境配置 |
| API 返回格式统一 | `{code, data, message}` |

### 2.4 敏感信息约束

**❌ 禁止**将以下信息写死到代码中：

- 密码、API Key、Token
- 数据库连接字符串
- 云服务密钥
- 私钥、证书
- Session/Cookie 密钥

**✅ 必须通过**环境变量或配置文件读取。

## 三、提交约束

### 3.1 提交前强制检查

在 `git commit` 之前，必须执行以下检查：

```
❓ 是否所有改动与需求相关？            → 是/否（否则撤销）
❓ 是否有遗留的调试代码？              → 否
❓ 是否有硬编码的敏感信息？            → 否
❓ 提交信息是否符合格式？              → type(scope): message
❓ Python 语法是否正确？              → python3 -c "import ast; ast.parse(open('file').read())"
```

### 3.2 提交信息格式

```
<type>(<scope>): <简短描述>

示例:
  feat(vben): 合同列表页添加搜索功能
  fix(backend): 修复 stage-payment 关联错误
  docs(workflow): 更新自动化流程文档
  ci: 更新 GitHub Actions 配置
  refactor(api): 重构 API 请求封装
```

## 四、CI 约束

### 4.1 GitHub Actions CI 检查项

每次 push 到 master 时自动执行：

1. **Python Lint**（ruff + 语法检查）
2. **UI 文件完整性**（API/View/Route 文件存在）
3. **提交信息格式**（仅 master）
4. **路由规范检查**（`path: ''` 验证）

### 4.2 CI 失败处理

如果 CI 失败：

1. **停止**当前工作
2. **查看** CI 日志：`https://github.com/Samueljackson6/pm-director/actions`
3. **修复** CI 指出的问题
4. **重新推送**修复后的代码

## 五、部署约束

### 5.1 部署流程

```bash
# 标准流程（推荐）— 自动触发
git push origin master
# → GitHub Actions 自动: Lint → UI验证 → 部署到236 → 验证 → 审计

# 或手动在236上执行
ssh 236 "cd /home/samuel/.openclaw/workspace/pm-director && bash scripts/deploy-236.sh all"
```

### 5.2 部署前检查

- [ ] CI 通过（GitHub Actions 显示绿色）
- [ ] 本地代码已推送到 GitHub 最新
- [ ] 236 可 SSH 连接
- [ ] 236 git 仓库没有冲突

### 5.3 部署后验证

必须执行验证脚本：

```bash
bash scripts/deploy-236.sh verify
```

预期结果：
```
Frontend: HTTP 200
Backend:  HTTP 200
Contracts JS: 2 files
Invoices JS:  1 file
Suppliers JS: 1 file
```

## 七、服务管理约束

### 7.1 236 systemd --user 服务

| 服务 | systemd 名称 | 端口 | 管理命令 |
|------|-------------|------|---------|
| 后端 | pm-backend | 8800 | `systemctl --user restart pm-backend` |
| 前端 | pm-frontend | 5777 | `systemctl --user restart pm-frontend` |

**禁止**使用 `nohup` + `pkill` 或 `setsid` 管理服务。必须用 systemd --user。

服务文件位置：`~/.config/systemd/user/pm-backend.service`

### 7.2 服务异常处理

如果服务挂了：

```bash
# 检查状态
sudo systemctl status pm-backend
sudo systemctl status pm-frontend

# 查看日志
sudo journalctl -u pm-backend -n 50 --no-pager
sudo journalctl -u pm-frontend -n 50 --no-pager

# 重启
sudo systemctl restart pm-backend
sudo systemctl restart pm-frontend
```

## 八、Vben 框架铁律（2026-07-02 强制执行）

**⚠️ 任何对 Vben 框架的操作——开发、维护、配置、部署、调试——必须先查阅 `docs/vben-official-index.md` 找到对应官方文档章节，再动手。** 这是不可违反的底线。

| 规则 | 说明 | 违规后果 |
|------|------|---------|
| **查文档再改代码** | 所有 Vben 相关活动前，先读 `docs/vben-official-index.md` 找到官方链接 | Code Review 拒绝 |
| **禁止凭猜测修改** | 不确定的必须查文档或向用户确认，不可自行推断实现 | 回滚 |
| **后端接入按官方格式** | `get-permission-info` 菜单数据遵循 Vben 官方规范 | 登录/菜单异常 |
| **变更前后必须验证** | curl 验证 API + 浏览器清缓存 + Console 无报错 | 视为未完成 |
| **文档不足时搜社区** | GitHub Issues/Discussions/Playground 示例 | 跳过可能错过已知方案 |
| **部署配置前读部署文档** | 构建、Nginx、环境变量等先看官方部署说明 | 构建失败/部署异常 |

**完整规范见：** `docs/Vben开发规范.md` 第七章 + `docs/vben-official-index.md`
