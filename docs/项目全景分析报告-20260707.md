## 一、项目概述

### 1.1 定位与主要功能

pm-director 是一套**面向国网合同（科研类 / 服务类为主）的全生命周期管理系统**，当前已上线运行、持续迭代。核心能力：

- **合同识别与入库**：OCR（PaddleOCR + PP-StructureV3）识别合同 PDF/.docx，提取结构化字段，人工校验后入库。
- **合同台账**：合同列表、详情、分类（科研/服务）、多维筛选（金额、签订日期、甲方、状态）。
- **财务管理**：客户开票/回款跟踪、供应商开票/付款跟踪、应付账款账龄。
- **项目进度**：研究阶段（stages）、付款计划（payments）、交付物清单（deliverables）的联动视图。
- **供应商管理**：供应商的合同/开票/付款画像。
- **数据同步**：飞书多维表格（主数据源）→ 本地 SQLite（镜像 + 增强），每周增量追加。
- **周报生成**：从飞书主表读取、按固定模板生成的 HTML 周报。

### 1.2 技术栈（已对齐真实代码）

| 层 | 技术 | 事实说明 |
|----|------|---------|
| 后端 | Python 3.11 + FastAPI + Uvicorn | 极简依赖，仅 `fastapi`、`uvicorn[standard]` |
| 数据库 | **SQLite**（`database/project_management.db`，`sqlite3` 直连） | **当前唯一真实存储**；MySQL 仅为规划迁移目标 |
| 前端框架 | **Vben Admin 5**（Vue 3 + Vite 7 + pnpm monorepo） | 框架源码在 `ui-vben/`，生产入口为 `ui-vben/apps/web-antd` |
| 业务代码 | `ui/`（api / views / router + `.env*`） | **构建时 COPY 进 `ui-vben/apps/web-antd/` 覆盖对应目录**，是业务逻辑唯一入口 |
| OCR | PaddleOCR + PP-StructureV3（本地部署） | 合同识别工具 |
| 外部系统 | 飞书多维表格 API | 财务数据源头 |
| 实验应用 | `ui-dashboard/`（独立 Vite + Vue3 + ECharts） | 非 vben、非主产品，疑似早期独立看板原型 |

> ✅ **关于「前端是否已全部 vben 化」的结论**：是。主前端就是 Vben Admin 5（`ui-vben/apps/web-antd`）。`ui/` 目录**不是**另一个前端框架，它只是业务代码（无 `package.json`，无法独立构建），在 `Dockerfile.frontend` 与 236 的 `deploy-236.sh` 中都被合并进 vben 的 `web-antd` 一起构建。不存在「双前端框架」并存的情况。

### 1.3 架构设计

```
┌─────────────────────┐        ┌──────────────────────┐        ┌──────────────────────┐
│   飞书多维表格(源)   │──同步──▶│  SQLite 本地库(镜像)  │◀───────│  后端 FastAPI(8800)   │
└─────────────────────┘        └──────────────────────┘        └─────────┬────────────┘
                                                                           │ /api /admin-api /system
                                                                           ▼
                                          ┌────────────────────────────────────────────┐
                                          │ Vben Admin 前端（构建自 ui-vben + ui 覆盖层）│
                                          │  - 本地 Docker：nginx :8900 → /web/         │
                                          │  - 236 老环境：python http.server :5777      │
                                          └────────────────────────────────────────────┘
```

- **分层**：数据层（SQLite） → 后端 API（`backend/routers/*`） → 前端 SPA（`ui-vben/apps/web-antd`，由 `ui/` 覆盖业务代码）。
- **响应约定**：统一包装为 Vben 格式 `{ code: 0, data: ..., message: "success" }`（见 `backend/models.py`）。
- **数据流**：飞书 → SQLite 每周增量追加（`batch_id` 可追溯，不覆盖历史）；OCR .docx → 结构化 JSON 缓存 → 入库。
- **前端构建本质**：`ui-vben` 提供 Vben 框架外壳，`ui/` 提供业务 api/views/router；二者在构建期合并，产物为 `apps/web-antd/dist`。

**数据模型（7 表 + 4 视图）**：

### 1.5 ⚠️ 关键澄清（新人必读，避免踩坑）

1. **前端 = Vben，且已全面 vben 化。** `ui/` 是业务代码 overlay，不是独立前端；`ui-vben` 是提交的 vben 框架。二者构建期合并。**没有「双框架并存」**。
2. **pm-director 当前不依赖 MySQL/Redis。** `backend/database.py` 明确 `sqlite3.connect` 本地文件；全仓库搜索无任何 MySQL/Redis 连接代码。规划中的 MySQL 迁移脚本 `migrations/mysql_migration_*.sql` **尚未执行**。
3. **`ruoyi-office-vben` 不是 Java 项目。** 它是 Vben Admin 5 的一个**外部 JS fork**（github.com/yuqing2026/ruoyi-office-vben，约 2.5GB，**不在本仓库 git 内**），仅用于 **236 老环境的 systemd 部署**——把 `ui/` 覆盖代码 copy 进去再 `pnpm build`。新方向（Docker 部署）改用**仓库内 `ui-vben`**，不再依赖这个外部克隆。两者不要混淆。
4. **前端 jiti 构建问题已在 Docker 路径解决。** `@vben/vite-config` 会把 `jiti` 内联进浏览器 bundle 导致 SPA 初始化报错；`Dockerfile.frontend` 已用 esbuild 编译 vite-config + 后处理替换 `createJiti` 为安全桩。`本地开发`（`pnpm dev`）天然绕过该问题。
5. **`ui-dashboard/` 是独立实验应用**，非 vben、非主产品，请勿当作主前端入口。

### 2.4 测试框架与用例执行 ❗（当前空白）

**现状：项目目前没有任何自动化测试。**
- 全仓库搜索 `test_*.py` / `*_test.py` / `pytest.ini` / `conftest.py` → 均无结果。
- 后端无 pytest/unittest；前端无 vitest/jest。
- CI（`ci.yml`）只做三类静态检查，**不运行任何测试**：
  1. Python Lint：`ruff check backend/` + `ast.parse` 语法校验；
  2. UI 文件完整性：检查 `ui/src/...` 关键文件存在、`path: ''` 路由格式；
  3. 提交信息格式：`type(scope): message` 校验。

**建议**：优先补齐后端接口测试（`pytest` + `fastapi.TestClient`），并入 CI 作为回归护栏。

### 2.5 ⚠️ 本地 Docker 环境的已知歧义（已记录，待修复）

- **compose 与 Dockerfile.frontend 未完全打通**：`docker-compose.yml` 的 `frontend` 服务使用 `image: nginx:alpine` + 挂载宿主 `ui-vben/apps/web-antd/dist`，**未用 `Dockerfile.frontend` 构建**。`Dockerfile.frontend` 是「全镜像构建（含 jiti 补丁）」路径，目前需手动 `docker build -f Dockerfile.frontend`。`docker compose build` 实际只重建了 backend。
- **由此推论**：CI 自托管 Runner 跑 `deploy-docker.sh` 时，`rsync` 同步的是源码，**不会自动构建前端 dist**；若宿主 `dist` 未更新，docker 前端会服务旧产物。需要时在 236 上手动 `pnpm build` 或改用 `Dockerfile.frontend` 镜像。

## 四、关键风险与改进建议（给团队）

1. **测试体系缺失（高）**：无 pytest/vitest，CI 不跑测试。优先补齐后端接口测试并入 CI。
2. **两套部署并存、口径不一（高）**：236 老环境（systemd + 外部 `ruoyi-office-vben` + 5777）与目标（Docker + 库内 `ui-vben` + 8900/`/web/`）并存，文档与脚本互相打架。应锁定「全 Docker」为唯一目标，废弃 `ruoyi-office-vben` 路径。
3. **CI 验证端口 BUG（中）**：`ci.yml` / `deploy-docker.sh` 验证用 `5777`，与 Docker 实际 `8900`/`/web/` 不符，会导致审计误判或依赖老环境残留。随 Docker 化修正。
4. **compose 与 Dockerfile.frontend 未打通（中）**：`docker compose build` 不重建前端 dist。建议让 `frontend` 服务直接 `build: { dockerfile: Dockerfile.frontend }`，或将 `pnpm build` 纳入 `deploy-docker.sh`。
5. **SQLite 单机风险（中）**：无并发控制、无自动备份、无主从。迁移 MySQL 前应建立 `database/` 定期备份。
6. **监控空白（中）**：无集中日志、无 APM、无告警。
