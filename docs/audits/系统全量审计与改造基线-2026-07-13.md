## 1. 管理结论

当前系统的问题不是单一的“界面不好看”，而是四套基础关系同时失控：

1. **菜单与路由没有唯一真值**：后端模式被改造成静态与动态路由混合，旧页面仍可直达；
2. **源码、容器和实际页面版本不一致**：PR 已合并不等于前端已部署，供应商模块至少存在三套版本；
3. **数据契约没有被数据库约束**：外键关闭、167 条外键违规、ID 与金额单位并存；
4. **GitHub 合并与发布门禁失效**：多个 PR 在 CI 完成前约 10 秒内合并，失败后仍留在 `master`，部署持续被跳过。

因此，继续逐页“美化”只会把新界面叠在旧结构上。当前应先停止扩展功能，按“安全与发布基线 → 路由和导航 → 数据契约 → 信息架构与 UI”的顺序整改。

系统级判断：**现有功能组件可局部保留，但系统关系和交互架构需要重设计，不适合继续零散修补。**

## 4. 用户现象的根因解释

### 4.1 “改了很多，但完全没看到效果”

已确认是发布漂移，不是主观感受：

- 当前前端容器仍运行旧构建；
- 前端容器代理的 `18080` 后端返回旧供应商组件；
- Windows 原生 `8800` 进程返回新版供应商组件；
- 磁盘 `backend/routers/auth.py` 又与原生进程不同；
- GitHub 多个相关 PR 的部署 Job 被跳过。

也就是说，同一台机器同时存在“新后端进程、旧容器后端、旧前端产物、旧磁盘菜单定义”。用户刷新浏览器无法解决镜像和产物未更新问题。

### 4.2 “详情返回后去了没看过的页面”

已实测和源码双重确认：

- 项目详情固定返回 `ProjectList`，浏览器落到 `/projects`；该页面不在后端菜单，但静态路由使其可访问；
- 客户发票详情固定返回旧 `InvoiceList`，而不是 `CustomerInvoices`；
- 供应商详情固定返回 `SupplierList`，当前解析成 `/suppliers//suppliers` 并 404；
- 看板告警卡父子组件会对一次点击分别导航，且目标不一致。

根因不是浏览器历史偶发异常，而是页面没有统一“来源与返回契约”，并且旧路由仍在运行时存在。

### 4.3 “菜单结构和关联逻辑不清楚”

当前菜单只展示 5 个一级模块，但代码中还存在项目、旧发票和大量上游模块路由。前端启动时无条件加载模块路由，再和后端菜单合并，导致：

- 菜单不可见不等于路由不可达；
- 同一路由名称可能对应两个路径；
- 同一业务同时存在旧页和新页；
- 权限、菜单、浏览器 Tab 和返回按钮使用不同事实源。

## 5. 系统设计合理性评估

### 5.1 合理且应保留的部分

- Vben + Vue Router + FastAPI + SQLite 的技术组合足以支撑当前规模，无需为改造更换技术栈；
- 新综合看板已经形成合同、开票、回款、项目和风险的管理视角；
- `StateBlock` 开始统一加载、错误和空态；
- 客户财务与供应商财务有较清晰的模块边界；
- 部分详情页已经采用紧凑 KPI、分区卡片和关联对象入口；
- 数据库已有 `contract_project_link`、`contract_id_mapping` 等桥接设计雏形。

### 5.2 不合理的系统性问题

#### A. 路由职责反转

项目规则要求后端菜单是真值，但底层生成器主动合并所有静态业务路由。这使静态“兜底”变成长期并行系统，权限控制无法证明完整。

#### B. 业务对象边界不清

合同、项目、发票、回款、付款和供应商之间同时使用 `contract_id`、`project_id`、`ZH02*`、`SGSC*`。部分表字段名说“项目”，实际外键指向“合同”。页面也把项目编号当合同详情 ID。

#### C. 兼容层没有退役策略

旧 `/invoices`、旧供应商页和旧列表路由不是单向重定向，而是仍被新页面主动引用。兼容层因此从临时迁移工具变成新的依赖。

#### D. 运行环境不是可重现状态

本机同时运行原生后端和容器后端，文档端口又指向无监听的 `8900`。没有统一版本指纹、部署后验收和回滚证据，无法回答“现在用户看的是什么版本”。

#### E. 数据库只存数据，不守契约

SQLite 外键功能关闭，应用层迁移后没有执行完整性校验。桥接表有数据但全部未验证，导致页面只能不断增加兼容查询和启发式映射。

### 6.1 当前规模

| 对象 | 数量 |
|---|---:|
| 合同 | 45 |
| 项目 | 50 |
| 发票 | 87 |

### 6.2 完整性问题

`PRAGMA foreign_keys` 返回 `0`。`pragma_foreign_key_check` 共 167 条违规：

| 子表 | 父表 | 违规数 |
|---|---|---:|
| `contract_clauses` | `contracts` | 66 |
| `stage_payment_link` | `payments` | 38 |
| `stage_payment_link` | `stages` | 34 |
| `contract_type_attributes` | `contracts` | 11 |
| `project_status` | `contracts` | 11 |
| `contract_project_link` | `contracts` | 5 |
| `finance_records` | `contracts` | 1 |
| `supplier_contracts` | `contracts` | 1 |

这意味着“页面能查到数据”不能证明数据关系正确。启用外键前必须先备份、分类和确认孤儿记录，不能直接删除或自动改写。

### 6.3 ID 与映射

- 合同主键：39 个 `ZH02*`、6 个 `SGSC*`；
- 映射表：32 条；
- `last_verified`：32 条全部为空；
- 合同项目关联：5 条找不到父合同；
- 供应商合同：1 条找不到父合同。

建议先定义不变的内部主键，再把 `ZH02`、`SGSC`、财务编号作为可验证的外部标识，不再让业务编码直接承担所有表的主键职责。

### 6.4 金额单位

当前合同金额主要按“万元”存储，发票和回款按“元”录入，但旧发票页和客户财务页直接把原值标为“万元”。浏览器实测旧发票页显示：

- 累计开票 `11,557,354.57 万元`；
- 累计回款 `6,596,706.97 万元`。

数据库相同聚合值实际是“元”，正确展示应先除以 10,000。综合看板又显示约 `1,197.09 万元`、`669.67 万元`，两处口径不同，用户无法判断哪一个可信。

金额字段必须在 API schema 中明确单位，前端禁止仅靠列标题约定。

## 7. UI/UX 与设计审计

完整评分见仓库根目录 `DESIGN-IS-2026-07-13/`。本次评分为 **7/30，结论 REDESIGN**。

这里的“重设计”不是推倒技术栈，而是重做用户可见的信息架构、导航、页面关系和视觉规范；现有可用业务组件可以迁移。

### 7.1 当前视觉问题

- 顶部和左侧同时承担菜单，内容区又有浏览器式 Tab，三层导航竞争；
- 看板单屏信息密度过高，指标、告警、图表和表格没有清晰的主次任务；
- 31 种引用色、26 种当前渲染色，状态色口径不够克制；
- 主要文本最低对比度约 `2.16:1`；
- 新旧页面样式并存：现代卡片页、旧大表格、404、空重定向组件共同出现；
- 多处使用 Emoji、渐变、圆角卡片和大数字，但没有统一设计 Token 约束；
- “综合驾驶舱”“全域总览”“Top 客户”“综合评级”等标签大于实际数据能力。

### 7.2 可用性与诚实性问题

- 合同列表“导出”只提示开发中；合同详情“导出”无事件；
- 供应商“查看全部”无点击处理；
- 供应商“风险优秀”和“综合评级”来自简单启发式评分，却没有说明口径；
- “暂无任何风险记录”只能证明当前扫描因子没有命中；
- `active` 状态在不同位置分别叫“进行中”和“执行中”；
- 发票概念把“客户回款”作为发票类型，业务语义混在一起。

### 7.3 性能与注意力成本

- 初始 JavaScript：约 4.07MB；
- 主视图请求：87 个；
- 最大单个 JS：约 3.61MB；
- 一次冷启动 60 秒内未达到可交互条件，并出现 10 秒 API 超时；
- 初始页面没有强制弹窗和持续动画，这一点可以保留。

### 7.4 重设计边界

优先重设计：

1. 全局导航与信息架构；
2. 列表 → 详情 → 关联对象 → 返回的主流程；
3. 金额、状态、风险、进度的统一表达；
4. 表格和详情页的视觉 Token 与状态规范；
5. 旧路由和旧页面的迁移/退役策略。

暂不重做：

- FastAPI、Vue/Vben 技术栈；
- 已验证有效的 API 能力；
- 综合看板已有的核心指标定义，待数据口径确认后复用；
- 数据清洗中需要业务确认的记录。

### 8.1 当前问题

#### 分支保护为空

`master` 当前没有 Branch Protection，也没有 Repository Ruleset。因此 PR、审查和 CI 都不是合并前置条件。

#### PR 在检查结束前合并

| PR | 创建到合并 | 合并后检查 | 部署 |
|---|---:|---|---|
| #29 供应商财务/发票/回款 | 约 11 秒 | Python Lint、Backend Test 失败 | 跳过 |
| #30 发票详情重定向 | 约 10 秒 | Python Lint、Backend Test 失败 | 跳过 |
| #31 综合看板 | 约 12 分钟 | 检查通过 | 跳过 |
| #32 供应商详情重构 | 约 12 秒 | Backend Test 失败 | 跳过 |

PR #32 的失败是新增 `supplier_contacts` 查询后没有更新测试桩，属于该 PR 直接造成的回归，不应视为无关基线失败。

#### 开放 PR 已过期

PR #25 从 2026-07-08 起未更新，仍修改旧 `views/invoices` 和 `views/suppliers` 页面，与后来 #29–#32 重叠，Backend Test 失败。建议关闭为 superseded，或重新开一个只解决当前规范路由的 PR；不建议直接 rebase 后合并全部旧改动。

#### Issue 与真实交付脱节

- #10 “根除 stale-dist”仍开放，且当前正发生 stale-dist；
- #8/#9 仍显示看板后端/前端待做，但 #31 已合并；
- #22–#24 已关闭，但供应商页面运行验收失败；
- 本地 `.github/issues/issue-4...` 的编号和 GitHub 真正 Issue #4 含义不同，容易错误引用。

#### CI 名称与实际能力不一致

`UI Validate` 检查旧目录 `ui/src`，缺失时只打印 `MISSING`，不会失败。主前端 `ui-vben` 没有被该 Job 验证。

CI 还缺少：

- 主前端 lint、typecheck、build；
- 菜单与路由矩阵 E2E；
- 详情返回 E2E；
- SQLite 外键完整性检查；
- 迁移 dry-run 和备份校验；
- 构建 SHA/静态资源指纹校验；
- 部署后的页面/API/版本一致性验收；
- 自动回滚或人工批准环境。

### 8.2 推荐协作顺序

以后每项改动按以下顺序执行：

```text
Issue 定义问题和验收标准
  → 设计/RFC 确认路由、数据和 UI 边界
  → 从最新 master 建短分支
  → Draft PR，先展示方案与风险
  → 实现 + 单元/契约/E2E/迁移 dry-run
  → 至少 1 人审查并解决全部对话
  → 所有必需检查完成且通过
  → 合并 master
  → 受保护环境部署
  → 版本指纹 + API + 页面烟测 + 回滚检查
  → 验收通过后关闭 Issue
```

禁止顺序：先合并、等 CI；先关 Issue、再验证部署；一个 PR 同时改路由、数据、UI 和基础设施。

### 8.3 建议的 GitHub 门禁

`master` 至少启用：

- 禁止直接 push 和 force push；
- 必须通过 PR；
- 至少 1 个批准审查；
- 所有 Review Conversation 必须解决；
- 分支必须为最新；
- 必需检查：Python Lint、Backend Test、Frontend Lint、Typecheck、Unit Test、Build、Route E2E、Data Integrity；
- 生产部署使用 GitHub Environment 人工批准；
- 合并后部署成功才自动关闭交付 Issue。

## 9. 分阶段改造顺序

### 阶段 0：冻结与安全止血（1–2 天）

- 暂停 UI 新功能 PR；
- 轮换并移除疑似泄露凭据，检查 Git 历史；
- 保护 `master`；
- 备份数据库并记录校验值；
- 明确唯一运行入口和当前部署 SHA；
- 关闭或重定义 PR #25。

验收：无法再合并失败 PR；运行页面显示的 SHA 与 `master` 部署目标一致。

### 阶段 1：路由与发布基线（3–5 天）

- 决定真正使用 `backend` 或官方 `mixed`，不再名实不符；
- 建立唯一菜单/路由清单；
- 修复供应商绝对子路径和组件版本；
- 旧 `/invoices`、`/projects`、旧供应商页改为单向迁移或正式纳入菜单；
- 实现统一返回契约；
- 添加路由矩阵 E2E；
- 修复 stale-dist 和部署版本指纹。

验收：所有菜单点击、详情进入、返回和旧路径处理都有自动化证据。

### 阶段 2：数据契约与清洗（1–2 周）

- 定义内部主键、外部业务编号和映射状态；
- 明确金额最小单位；
- 定义合同/项目/发票/回款/付款状态字典；
- 将 167 条外键违规分类为可自动修复、需业务确认、应归档；
- 先迁移 dry-run，再逐步启用外键；
- 对映射加入验证人、验证时间和来源。

验收：外键检查为 0；金额单位测试覆盖 API 和 UI；所有不确定数据均有业务确认记录。

### 阶段 3：信息架构与 UI 重设计（1–2 周）

- 先做菜单树、用户任务流和低保真原型；
- 只保留一套全局导航；
- 定义列表/详情/关联/返回规范；
- 建立集中式颜色、间距、字体、状态和数字格式 Token；
- 先重做一个纵向切片，例如“合同列表 → 合同详情 → 项目详情 → 返回”；
- 经真实用户首击测试后再扩展到财务和供应商。

验收：主要任务不经过隐藏旧页面；视觉和交互 Token 可被自动检查；WCAG AA 对比度通过。

### 阶段 4：逐模块交付与退役（持续）

- 合同与项目；
- 客户财务；
- 供应商与供应商财务；
