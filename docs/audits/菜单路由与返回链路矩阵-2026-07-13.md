# 菜单、路由与返回链路矩阵

审计日期：2026-07-13  
审计范围：本地 `master`、Windows 原生后端 `8800`、Docker 后端 `18080`、Docker 前端 `18090/web/`  
审计性质：只读；未修改业务代码和数据库

## 1. 结论

当前系统没有单一、可靠的菜单与路由真值。

名义配置是 Vben `backend` 路由模式，但运行逻辑会把前端静态路由和后端动态路由合并。加上原生后端、容器后端、磁盘源码和前端构建产物版本不一致，最终形成至少四层漂移：

1. 后端权限菜单；
2. 前端 `routes/modules` 静态路由；
3. 页面组件内部的命名路由或硬编码路径；
4. 用户实际访问的旧构建产物。

因此，当前不能把“源码里有页面”理解为“菜单能访问”，也不能把“PR 已合并”理解为“用户已经看到”。

## 2. 路由真值冲突

| 层级 | 声明 | 实际 | 影响 |
|---|---|---|---|
| 项目规则 | 后端菜单是唯一真值，前端模块只兜底 | 规则见 `docs/vben-framework-rules.md:13-17` | 设计方向本身合理 |
| Vben 配置 | `accessMode: backend` | 官方后端模式应由接口动态生成路由 | 应只保留一套业务路由事实源 |
| 路由生成器 | 后端动态路由 | `generate-routes-backend.ts:55-56` 合并静态和动态路由 | 实际变成未声明的混合模式 |
| 静态路由入口 | 业务模块只作兜底 | `routes/index.ts:7-16,36` 无条件加载全部模块 | `/projects`、`/invoices` 等旧页面仍可直达 |
| 运行菜单 | 应与磁盘源码一致 | `8800`、`18080`、磁盘 `auth.py` 的供应商定义互不一致 | 同一菜单在不同入口落到不同组件 |

## 3. 当前一级菜单

| 一级菜单 | 后端路径 | 当前可见子菜单 | 隐藏详情节点 | 主要问题 |
|---|---|---|---|---|
| 概览 | `/dashboard` | 综合看板 | 分析页 | `分析页` 标记不可见，但浏览器侧栏仍实际显示 |
| 合同管理 | `/contracts` | 合同列表 | 合同详情 | 静态列表是 `/contracts`，后端列表是 `/contracts/list`，名称相同路径不同 |
| 客户财务 | `/customer-finance` | 客户发票、客户回款 | 发票详情、回款详情 | 客户发票列表和详情仍引用旧 `InvoiceDetail`、`InvoiceList` |
| 供应商管理 | `/suppliers` | 供应商列表 | 供应商详情 | 当前菜单路径 404；旧路径生成 `/suppliers//suppliers` |
| 供应商财务 | `/supplier-finance` | 供应商发票、供应商付款 | 发票详情、付款详情 | 结构相对一致，但仍受全局双路由体系影响 |

当前后端菜单不包含“项目管理”和旧“发票管理”，但这两个模块仍可直接访问。

## 4. 页面关系矩阵

### 4.1 概览

| 项目 | 后端动态定义 | 前端静态定义 | 浏览器实测 | 判断 |
|---|---|---|---|---|
| 综合看板 | `/dashboard/overview/index` → `/dashboard/overview/index.vue` | 相同 | 正常渲染“综合驾驶舱” | 当前有效，但标题与菜单名不一致 |
| 旧综合看板地址 | 无 | 无明确业务定义 | `/dashboard/overview` 为空壳 | 应添加显式重定向或下线 |
| 分析页 | `visible: false` | 上游页面仍存在 | 侧栏实际可见“分析页” | 隐藏语义没有正确落地 |
| 风险告警卡 | 目标为旧 `/invoices`、`/projects`、`/contracts` | 子组件再附加筛选参数跳转 | 未逐卡点击验收 | 父子同时 `router.push`，存在跳转竞争和历史污染风险 |

告警卡父子导航证据：

- 父组件映射：`dashboard/overview/index.vue:74-83`；
- 子组件映射与第二次跳转：`dashboard/overview/components/alert-strip.vue:37-79`。

### 4.2 合同

| 页面/动作 | 当前定义 | 目标 | 风险 |
|---|---|---|---|
| 后端合同列表 | `/contracts/list`，名称 `ContractList` | `views/contracts/index.vue` | 浏览器可渲染表格 |
| 静态合同列表 | `/contracts`，名称同为 `ContractList` | 同一组件 | 重名、不同路径；命名跳转结果取决于动态路由覆盖顺序 |
| 合同详情返回 | `router.push({ name: 'ContractList' })` | 合同列表 | 没有保存来源、筛选、分页和滚动位置 |
| 合同详情 → 发票 | `InvoiceDetail` | 旧发票详情重定向页 | 绕行旧路由，增加双页面耦合 |
| 合同详情 → 项目标题 | `ProjectDetail` + query | 项目详情 | 可用契约之一 |
| 合同详情 →“查看详情” | `ProjectDetail` + params | 项目详情 | 路径没有 `:id`，参数会丢失；与同卡片标题行为不一致 |

证据：`contracts/detail.vue:219,263,277,1303-1305`，`routes/modules/projects.ts:10-12`。

### 4.4 供应商

供应商模块同时存在三套定义：

| 来源 | 列表路径/组件 | 详情组件 | 状态 |
|---|---|---|---|
| 原生后端进程 `8800` | `list` → `/suppliers/list/index` | `/suppliers/detail/index` | 新版进程内定义 |
| 磁盘 `backend/routers/auth.py` | 绝对子路径 `/suppliers` → `/suppliers/index` | `/suppliers/detail` | 旧版；与当前 HEAD 一致 |
| 容器后端 `18080` | 绝对子路径 `/suppliers` → `/suppliers/index` | `/suppliers/detail` | 旧版；前端容器实际代理来源 |
| 前端静态路由 | 空子路径 → `views/suppliers/index.vue` | `views/suppliers/detail.vue` | 旧版 |
| 新页面文件 | `views/suppliers/list/index.vue` | `views/suppliers/detail/index.vue` | 文件存在，但当前前端产物不可达 |

浏览器实测：

- `/suppliers/list`：404；
- `/suppliers`：最终地址 `/suppliers//suppliers`，404；
- `/suppliers/detail?id=<有效ID>`：旧详情能直接打开；
- 点击旧详情返回：落到 `/suppliers//suppliers`，再次 404。

根因是后端父路径 `/suppliers` 下又使用绝对子路径 `/suppliers`，加上同名静态路由与旧构建产物共同作用。

## 5. 返回契约审计

当前详情页普遍使用“固定跳列表”，不是“返回来源”。

| 详情页 | 当前返回 | 是否在菜单 | 是否保留筛选/分页/滚动 | 结论 |
|---|---|---:|---:|---|
| 合同详情 | `ContractList` | 是 | 否 | 目标大体正确，体验状态丢失 |
| 项目详情 | `ProjectList` | 否 | 否 | 错误：返回不可见列表 |
| 客户发票详情 | `InvoiceList` | 否 | 否 | 错误：返回旧发票页 |
| 客户回款详情 | `CustomerReceipts` | 是 | 否 | 目标正确，状态丢失 |
| 供应商详情（旧/新） | `SupplierList` | 菜单定义损坏 | 否 | 实测返回 404 |
| 供应商发票详情 | `SupplierInvoices` | 是 | 否 | 目标正确，状态丢失 |
| 供应商付款详情 | `SupplierPayments` | 是 | 否 | 目标正确，状态丢失 |

推荐的统一契约：

1. 列表进入详情时携带 `from`、筛选、页码和滚动位置；
2. 详情页优先返回可信来源；
3. 没有可信来源时回到该模块唯一规范列表；
4. 所有旧路径只做单向重定向，业务页面不得再主动跳回旧路径；
5. 对返回目标建立 E2E 矩阵，而不是只检查页面是否能打开。

## 6. 建议的目标菜单

本轮不直接替用户决定最终信息架构，但当前最小、稳妥的目标结构应先收敛为：

```text
综合看板
合同管理
  └─ 合同列表
项目执行
  └─ 项目列表
客户财务
  ├─ 客户发票
  └─ 客户回款
供应商管理
  └─ 供应商列表
供应商财务
  ├─ 供应商发票
  └─ 供应商付款
```

详情页全部隐藏于菜单，但必须有明确的来源与返回契约。`分析页`、旧 `/invoices`、重复 `/contracts` 列表和两套供应商页面应在迁移完成后退役。

## 7. 验收基线

路由治理完成前，至少需要通过以下门禁：

- 每个业务路由只有一个规范名称、路径和组件；
- `backend` 模式不再无条件合并业务静态路由；
- 所有旧路径只能重定向到新路径，不能反向引用；
- 菜单可见项逐一点击为 HTTP/页面成功态；
- 隐藏详情逐一从列表进入并返回原来源；
- 直接访问孤立旧路径得到规范重定向或明确 404；
- 运行构建暴露提交 SHA，必须等于部署目标 SHA；
- 原生后端、容器后端和磁盘源码的菜单摘要必须一致。

## 8. 证据边界

- 浏览器验证针对本机 `18090` 当前运行构建，不代表远端 236/公网环境。
- 合同列表受异步加载波动影响，未稳定完成“列表点击详情再返回”的浏览器链路；对应问题由源码路由契约确认。
- 客户发票正确详情路径是 `/customer-finance/invoice-detail`；临时测试中另一个嵌套地址为空页，不把它计为正式路由缺陷。
- 数据库只读审计基于当前工作区快照；该文件在本任务开始前已有用户修改。
