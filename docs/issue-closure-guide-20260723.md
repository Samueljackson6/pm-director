## 二、Issue #57 评估 — 转 RFC

### 问题描述

`suppliers` 表同时存储甲方（合同甲方/业务单位）和乙方（实际供应商/承揽商），导致：
- 列表无法区分甲方/乙方筛选
- 台账同步时 party_a/party_b 为纯文本，无法通过供应商表关联
- 统计口径混乱（甲方回款 vs 乙方付款逻辑不同）

### 现状分析

| 维度 | 当前状态 | 影响 |
|------|---------|------|
| suppliers 表字段 | 无 category 标识 | 无法区分甲方/乙方 |
| 甲方数据 | 22条（客户单位），占 active 记录 40% | 混在供应商表中 |
| contracts.party_a | 纯文本，不关联 suppliers | 台账同步失败 |
| contracts.party_b | 关联 suppliers | 部分正确 |
| 匹配时混淆 | 甲方名称可能匹配到乙方 | 数据错误 |

### 推荐方案：RFC（Request for Comments）

**转为 RFC 后，后续实现步骤：**

#### Phase 1：数据层（后端）
1. `suppliers` 表新增 `category` 枚举：`client` / `vendor`
2. 迁移现有数据：根据 `party_a`/`party_b` 上下文分类
3. `contracts` 表新增 `party_a_supplier_id` 外键（可选）

#### Phase 2：API 层
4. `/api/suppliers` 增加 `?category=client|vendor` 过滤参数
5. 台账同步逻辑区分甲方/乙方处理

#### Phase 3：前端层
6. 供应商列表增加 Tab 切换：甲方客户 / 乙方供应商
7. 台账页面区分展示

### 工作量评估
- 数据迁移：S（约 2-4 小时）
- API 修改：M（约 1-2 天）
- 前端改造：M（约 1-2 天）
- 测试验证：S（约 0.5 天）
- **总计预估：3-5 天**

### 建议
- 先由业务方确认：甲方是否真的需要作为"供应商"管理？
- 如果甲方只是合同参考信息，可以考虑从 suppliers 表移除，保留在 contracts.party_a 即可
- 如果甲方也需要管理（联系人、资质等），则按上述方案实施
