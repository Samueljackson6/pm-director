## 关键问题

1. 国网客户合同有哪些稳定类型、付款节点、开票条件和回款例外？
2. 客户合同与供应商采购合同、设备、服务、发票、付款如何关联？
3. 合同原件、DOCX、Excel、数据库和人工补录分别是什么可信度？
4. 当前 Vben 版本能直接承载哪些页面与组件，哪些 Demo 交互需要调整？
5. Web、小程序和 App 各自服务谁、处理什么任务、共享哪些 API 与领域对象？

## 已确认决策

| 决策 | 理由 |
|------|------|
| 新 Demo 直接在真实 Vben 页面中实现评审模式 | 组件兼容必须由真实 Page/VxeGrid/AntDV/主题令牌和路由运行结果证明 |
| 当前主入口改为合同档案与数据核验工作台 | 与合同原件、低质量表格、人工核验并存的数据成熟度一致 |
| 科技感集中在证据轨道、关系图和轻量状态动效 | 默认浅色，避免深色大色域，同时保留可识别的专业设计感 |
| Demo 使用只读快照与脱敏业务样例并显式标注来源/推断 | 保证可评审且不把模拟、推断或历史快照伪装成实时事实 |
| 后端先只读评审，生产改造列入评审通过后的实施批次 | 用户允许后端改造，但当前首要目标是设计评审 |
| 新 Demo 聚焦 4 个任务屏：档案总览、合同详情、数据核验、移动任务 | 先验证业务主轴和多端边界，不重复铺设脱离数据成熟度的大量台账页 |
| 旧 Demo 降级为视觉探索稿 | 用户确认其视觉、内容和业务逻辑均未达到评审标准，不再作为生产实施依据 |
| 四任务域和履约链降级为待验证假设 | 必须先用真实合同、收付款和供应商关系验证，不将先前设计判断当成业务事实 |
| 暗色大色域退出默认方向 | 后续以浅色工作区为主，暗色仅作为可选主题或局部强调 |
| Vben 兼容性必须有组件级证据 | 每个页面模块要对应现有 Vben/Ant Design Vue 组件、API 和官方文档 |
| 客户应收与供应商应付严格分轨 | 当前 inbound 语义冲突、供应商付款为 0，任何合并时间线都会误导 |

## 错误记录

| 错误 | 次数 | 处理方式 |
|------|------|----------|
| HTML 说明性注释触发 comment-checker | 1 | 删除非必要注释，改用页面内可见 Demo 标识 |
| Codex 内置 Python 缺少 Playwright | 1 | 不安装项目依赖，改查机器现有 Python/Node 浏览器运行时 |
| `with_server.py` 未收到 PowerShell 分离形式的 `--port` | 1 | 改用显式 `--server=...` 与 `--port=4173` 参数 |
| 移动端测试同名按钮触发 Playwright strict violation | 1 | 将定位器限定到供应商详情页容器 |
| Windows 下 `with_server.py` 未回收 `http.server` 子进程 | 1 | 精确清理 Demo 端口进程，改用隐藏进程并在 `finally` 回收 |
| 模糊清理命令匹配到承载命令的 PowerShell | 1 | 限定进程名为 `python.exe`，后续只回收 `Start-Process` 返回的 PID |
