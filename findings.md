# 阶段3发现与决策记录

> 2026-07-19：本文件作为调研和纠偏证据保留。接手团队应从 [阶段 3 交接入口](docs/handoff/README.md) 获取当前目标、进度、任务顺序和风险清单。

## 用户要求

- 开始阶段3任务和方案细化。
- 以可交互 Demo 形式提交相关页面评审。
- 遵循项目规范，同时加强科技、时尚、美观、丰富、实用与动效表达。
- 允许为了前端设计评审并改造后端。
- 将窄范围盘点任务委派给子代理，主代理负责评审、设计、实现与验收。

## 2026-07-15 用户评审纠偏

- 旧 Demo 的页面和组件尚未证明与当前 Vben 框架兼容。
- 页面视觉不够成熟，暗色大色域过多，应减少大块暗色布局。
- 页面内容、功能需求和业务逻辑与现状严重脱节。
- 必须先理解国网合同类型、合同内容、收款特点和供应商关联关系。
- 当前主要数据源仍是合同原件和质量较差的表格，系统处于逐步成熟管理的过渡期。
- 新设计必须考虑未来小程序和 App，而不能只做桌面后台页面。
- 旧 Demo 和既有信息架构结论均降级为待验证假设。

## 已知项目基线

- 目标信息架构为：驾驶舱、合同履约、客户应收、供应商履约。
- 驾驶舱保持单路由四视图：全局、经营、项目、财务。
- 详情页核心识别组件为“履约链”。
- Vben 项目使用后端路由模式，生产菜单需以后端权限路由为真值。
- 当前生产前端仍存在旧菜单、全局页签和部分错误跳转，不能视为阶段3完成态。
- 图谱确认后端已有项目详情、阶段、付款、交付物、进度等分拆接口，项目履约链可优先复用现有数据，不需要先造新聚合接口。
- 图谱确认合同、供应商、发票与仪表盘已有独立路由模块，但当前前端同时保留多套旧 UI/API 路径，生产接入时必须限定在 `ui-vben/apps/web-antd`。
- 阶段3现有设计文档与历史决策一致：驾驶舱必须保持单路由四视图，不扩展为多路由工作台。

## 技术决策

| 决策 | 理由 |
|------|------|
| 独立静态 SPA Demo | 快速评审、零生产依赖、便于浏览器自动化验证 |
| 原生 HTML/CSS/JavaScript | 不引入新依赖，不改变现有构建体系，离线即可打开 |
| 深海蓝导航 + 云雾白内容区 + 电光青蓝重点 | 兼顾企业后台可读性与科技时尚识别度 |
| 动效限制在 160–480ms，支持 reduced-motion | 保持反馈感并满足可访问性要求 |

以上技术决策仅记录旧 Demo 事实，不再代表新阶段3设计方向。

- Vben 本地规则确认：项目采用 `accessMode: 'backend'`，页面组件、`backend/routers/auth.py` 菜单和组件路径必须同步；列表优先 `useVbenVxeGrid`/VxeTable 适配层，图表必须用 `@vben/plugins/echarts`，请求必须走 `requestClient`。
- 本地官方索引确认可直接使用 `Page`、Vben Drawer/Modal/Form/VxeTable/Descriptions/TableAction；详情页可使用 Ant Design Vue 的 Card/Descriptions/Tabs/Timeline/Steps，但必须遵守现有适配层和设计令牌，不能另造应用壳层。
- 知识图谱和源码命中证明：生产合同详情已经使用 `StateBlock`，项目详情已有 `useVbenVxeGrid`、Descriptions、Steps、Timeline，Vben 主应用已有 Page/ECharts 示例；兼容性问题不是“组件能不能用”，而是旧 Demo 尚未按真实组件契约实现和验证。

## 当前审计假设

- 生产 Web 继续使用 Vben Admin 与 Ant Design Vue，但页面结构必须从现有组件能力反推，不接受静态 HTML 独立发明交互。
- 合同原件是合同条款与权利义务的源头真值；OCR、DOCX、JSON、Excel 和数据库字段需要标识来源和核验状态。
- 国网类项目可能存在里程碑、验收、开票、审计、质保金和分期回款的组合，不能用单一线性进度条代替。
- 供应商关系可能按客户合同、项目、采购合同、设备/服务、供应商发票和付款多层关联，需要用现有数据验证。
- 小程序和 App 应围绕移动场景任务设计，不复制桌面后台菜单。

## 国网合同业务基线（文档证据）

- `docs/深度需求分析与架构设计-v2.md` 记录的当时存量为 40 份合同、科研类 13 份、服务类 27 份；物资、施工、框架/协议库存是未来扩展类型，不能在当前页面中伪装成已有业务。
- 科研类核心产出是专利/论文/软著，按研究阶段和专家评审验收，付款通常跟里程碑绑定；服务类核心产出是报告/数据，付款可能是竣工+质保或分期。单一“签订→立项→部署→验收→回款”履约链无法表达两类合同差异。
- 当前至少存在内部项目编号、国网统一合同编号和财务编号三套编号，页面需要映射和检索，不应只显示一个虚构的 HT/XM 编号。
- 供应商既包括高校/科研机构，也包括科技、工程、测绘、劳务等主体；关联维度包含供应商合同金额、收票、付款、未开票、未付款，不能只做通用供应商健康度。
- 客户开票与供应商收票是方向相反的两条发票链：客户开票连接合同→收入/回款，供应商收票连接采购/成本→付款/抵扣；旧 Demo 对两者关系表达过浅。
- 旧数据中的 `invoice_history` 为非结构化文本，阶段与付款缺少关联，供应商信息分散，`project_status` 曾为空；任何仪表盘比例都必须带来源、批次和核验状态。
- 2026-07-01 文档曾把飞书主表定义为财务数据源头、DB 为镜像增强；用户本轮进一步强调合同原件和低质量表格仍是主要输入，因此需要按“合同条款真值/财务运营真值/系统派生”分域，而不是选一个全局唯一真值。
- 文档建议的优先业务视图是供应商合同/开票/付款、发票跟踪、合同类型细分、阶段-付款联动；这比旧 Demo 的通用 KPI 和风险卡片更接近实际需求。
- 科研合同通常是“研究阶段—考核目标—交付成果—付款条件”的 1:1、N:1 或少量 1:N 关系；只有合同明确提及、内容匹配或时间匹配时才可建立联系，时间匹配必须标为待人工确认。
- 服务合同以服务期、交付物、验收为主，常见 97% 验收款 + 3% 质保金、80%+17%+3%、80%+20%、50%+50% 等多种模式，不能预置单一节点模板；OCR 缺失、比例范围或条款不完整必须进入人工确认队列。
- 35 份国网技术服务合同样本显示：甲方为多个国网四川分支、乙方为本公司；这条“客户合同/应收”链必须与本公司向外部供应商采购形成的“供应商合同/进项票/付款”链严格分开，再通过项目或合同关系连接。
- 国网技术服务合同的违约、赔偿、保密、知识产权、违约金、逾期、解除/终止具有高覆盖率，结构化字段必须同时保留原条款片段、比例基数、触发条件和 OCR/人工核验来源。
- 对“合同写 100%，业务解释为 97%+3%”的情况，系统不得静默覆盖原条款；应同时记录原文结论、业务修正规则、修正人/时间及核验状态。
- 财务表格存在“完结/已开完”、汇总行、日期金额混排、跨省分类等口径；汇总值与明细和合同额不一致时应进入差异核对，不能用 UI 自动截断或冒充已确认。
- 现有合同详情审计证实：客户开票和客户回款曾混在 `invoices` 中，`payments` 更接近合同付款计划而非客户回款；把它们画成一条“付款/回款”线会直接产生错误指标。
- 当前详情数据仍存在 SGSC/ZH 双 ID、元/万元混存、映射缺失、阶段付款无 FK、财务快照与明细漂移等问题；新页面必须先显示 canonical ID、别名、单位、来源批次、差异和核验状态，再显示汇总 KPI。
- 2026-07-15 只读数据库复核：45 份合同（服务 32、科研 10、未分类 3），45 条合同—项目关联，32 条编号映射，55 家供应商，23 条供应商关系，84 条付款计划，43 条财务快照，供应商付款为 0，阶段—付款关联为 0。
- 当前 `contracts` 为 39 个 ZH 主键和 6 个 SGSC 主键；29 条填有 `sgsc_id`、`financial_id` 全空、`quality_bond_pct` 仅 1 条，证明统一编号、财务编号和质保比例都远未成熟。
- `invoices` 的 87 行中，53 行为 `outbound + 客户开票`，34 行为 `inbound + 客户回款`；这与旧“direction=inbound 即供应商发票”的开发计划直接冲突。当前 `direction` 不能单独决定业务语义，供应商发票必须另建/清洗模型后才能展示。
- 87 行发票/回款的 `supplier_id` 全空；79 行来自 `finance_excel_import` 且仍是 unverified。旧 Demo 的供应商发票匹配、已核验和付款闭环均没有数据事实支撑。
- `contract_type_attributes` 另有 29 条：服务 18、科研 8、物资 3，说明“法律/交易模板”与“项目执行形态”应拆为两条分类轴，不能只靠 `contracts.project_type`。
- 客户回款同时存在 34 条 `receipts` 与 34 条 `invoices(invoice_type='客户回款')`；`invoice_receipt_link` 只有 25 条，覆盖 23 张发票和 25 笔回款。新 UI 必须把“双模型并存/部分匹配”作为数据质量事实，不能呈现为已完整闭环。
- 84 条 `payments` 只有 1 条存在实际金额，证明该表主要是合同付款计划，不是客户回款或供应商实际付款。

## 项目架构与旧 Demo 诚实性证据

- `docs/项目全景分析报告-20260707.md` 明确主前端是 Vben Admin 5，生产入口为 `ui-vben/apps/web-antd`；`ui/` 是构建时覆盖层，`ui-dashboard/` 是早期实验应用。新 Demo 必须以 Vben 主产品为载体或建立逐组件映射。
- 当前真实存储是 SQLite；合同 OCR 流程为 PDF/DOCX→结构化 JSON→人工校验→入库，财务数据为飞书→SQLite 增量镜像。页面必须展示来源和核验状态，不能把数据库字段一律当成已确认主数据。
- 2026-07-07 报告记录当时自动化测试为空、Docker 与旧部署并存且 dist 可能过期；设计验收必须区分源码 Demo、Vben 开发态、构建产物和实际运行版本。
- 旧 Demo 文案存在直接诚实性冲突：页面写“经营数据已连接”“实时·2分钟”“实时口径”，README 又声明完全不连接 API。
- 旧 Demo 的导出、新建、标记完成、筛选、分页、通知、搜索等多项控件只显示占位 Toast 或根本无行为，视觉上却以真实操作呈现。
- 所有合同、项目和供应商行都进入同一硬编码详情；“逾期筛选”只改 URL/说明不筛数据；“6笔回款等待匹配”错误跳到供应商发票页。
- 项目和供应商详情复用了同一套客户合同履约链，证明旧 Demo 不是业务模型的视觉化，而是通用组件的重复套用。
- 旧 Demo 的健康度、同比、排序、自动匹配、企查查核验和高精度金额均无来源或计算规则；按 design-is 的诚实性标准，这是负载级缺陷。
- 2026-07-07 全景报告进一步确认 `ui-vben/apps/web-antd` 才是主产品，`ui/` 只是构建覆盖层，`ui-dashboard/` 是早期实验；独立静态壳不能证明生产兼容性。
- 当前持久化仍以 SQLite 为主、MySQL 为未来迁移目标；部署还存在 systemd/外部旧前端与 Docker/库内 Vben 两套口径，后续 Demo 验收必须标注“源码、构建物、实际运行时”三者版本。
- 业务证据优先读取：合同综合说明、科研/服务合同识别规则、条款规律、财务数据提取规范、供应商财务计划、无付款合同分析、合同详情审计与改造方案；合同详情审计已明确记录 KPI 矛盾、阶段付款映射编造、发票付款启发式匹配等高风险问题。

## Vben 原生 Demo 最终证据

- 评审路由 `/dashboard/overview/index?review=phase3` 已在当前 Vben 主应用中通过真实登录验证。
- `a-segmented` 在当前项目未全局注册；显式导入 Ant Design Vue `Segmented` 后，四屏切换由 Playwright 超时红灯转为通过。
- 静态三行数据使用 VxeGrid 时会因固定列产生重复行 DOM；验收改为核对三组唯一合同编号，避免把组件实现细节误判为数据重复。
- 静态只读表格关闭分页和刷新，避免出现“共 0 条”或暗示连接实时 API 的假操作。
- 最终验收：Vitest 3/3、ESLint、Stylelint、Vite 生产构建和 Playwright 浏览器流程全部通过。

## UX 审查结论

- 最小评审范围为驾驶舱、合同台账、合同详情、项目详情、客户应收代表页和供应商详情代表页。
- Demo 增加项目列表与供应商列表，用于完整验证目标菜单树和进入详情的路径。
- 唯一强视觉中心保持为“履约链”；禁止持续粒子、全屏脉冲、大面积霓虹和自动轮播。
- 关键交互必须验证 URL 状态、单次跳转、原路返回、移动端纵向履约链和 reduced-motion。
