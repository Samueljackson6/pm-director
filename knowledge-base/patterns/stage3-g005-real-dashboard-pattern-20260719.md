# 阶段 3 G005：真实驾驶舱的可解释下钻模式

> 记录日期：2026-07-19
>
> 适用范围：Vben 单路由驾驶舱、镜像数据和待核验业务域。

## 问题

驾驶舱容易把汇总值、推断状态和演示页面混在一起，造成“看起来可用、实际上不能处理”的界面：没有对象 ID、没有来源、没有筛选消费者，或切换视图时重复聚合请求。

## 可复用做法

1. 后端把动作建模为稳定契约：`object_id`、`reason`、`due_date`、`owner`、`status`、`target.path`、`target.query`。
2. 指标必须另有数据契约：口径、来源、覆盖范围、核验状态、数据截止时间；镜像数据默认待核验。
3. 正式产品页与评审 Demo 彻底解耦。Demo 可以保留，但不得以 Query 条件进入正式路由。
4. 同页视角只更新 hash Query。若切换不应重新取数，使用 `history.replaceState` 保持组件状态，浏览器测试断言聚合请求次数。
5. 空队列只能说明“未识别到可下钻记录”，不能说明“没有风险”或“数据已完成核验”。

## 验证要点

- 用只读连接执行驾驶舱 GET，并对 SQLite 文件计算前后哈希。
- 浏览器验证五视角、一个真实详情下钻、无重复聚合、无业务 4xx/5xx 与无未授权写请求。
- 将视觉和数据契约测试分开：源码边界测试防止 Demo 回流，接口测试固定字段，浏览器测试验证实际路由消费。
